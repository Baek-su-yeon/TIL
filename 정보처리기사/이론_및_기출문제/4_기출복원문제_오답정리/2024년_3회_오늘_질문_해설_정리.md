## 1번 — Java `String` 배열, 참조값, `equals()` 비교

### 핵심 개념

`String[] s = new String[3];`은 문자열 3개를 직접 저장하는 배열이 아니라, **String 객체를 가리키는 참조값 3개를 저장하는 배열**이다.

```java
static String[] s = new String[3];

s[0] = "A";
s[1] = "A";
s[2] = new String("A");
```

배열의 각 칸은 다음과 같은 역할을 한다.

```text
s[0], s[1], s[2] = 배열의 칸
각 칸 안의 값 = String 객체를 가리키는 참조값
```

즉, `s[0]`, `s[1]`, `s[2]`는 배열 인덱스로 접근하는 **칸의 위치**이고, 그 칸 안에는 각각 String 객체의 주소에 해당하는 참조값이 들어 있다.

---

### 문자열 리터럴과 `new String()`의 차이

```java
s[0] = "A";
s[1] = "A";
```

`"A"`처럼 코드에 직접 적은 문자열은 보통 **문자열 상수 풀(String Constant Pool)**에 저장된다. 따라서 `s[0]`과 `s[1]`은 같은 문자열 객체를 가리킬 수 있다.

```text
s[0] ─┐
      ├──> "A" 객체
s[1] ─┘
```

반면,

```java
s[2] = new String("A");
```

`new`를 사용하면 내용이 같아도 새로운 String 객체를 별도로 만든다.

```text
s[0] ─┐
      ├──> "A" 객체
s[1] ─┘

s[2] ───> 새 "A" 객체
```

따라서 `s[2]`가 가리키는 객체는 `s[0]`, `s[1]`이 가리키는 객체와 다를 수 있다.

---

### 주소가 다른데 `s[2]`로 접근 가능한 이유

주소가 다르다는 말은 `s[2]`라는 배열 칸이 없다는 뜻이 아니다.

```text
s[2]는 배열의 2번 칸이다.
그 칸 안에 들어 있는 참조값이 다른 String 객체를 가리킬 뿐이다.
```

즉, `s[2]`로 접근하는 과정은 다음과 같다.

```text
1. 배열 s의 2번 칸으로 간다.
2. 그 칸에 저장된 참조값을 읽는다.
3. 그 참조값이 가리키는 String 객체를 사용한다.
```

---

### `==`와 `equals()` 차이

| 비교 방식 | 비교 기준 | 예 |
|---|---|---|
| `==` | 참조값, 즉 같은 객체를 가리키는지 비교 | `s[0] == s[2]` |
| `equals()` | 문자열 내용이 같은지 비교 | `s[0].equals(s[2])` |

문제에서는 다음처럼 `equals()`를 사용했다.

```java
if(s[i-1].equals(s[i]))
```

`equals()`는 주소가 아니라 문자열 내용을 비교하므로,

```text
s[0].equals(s[1]) → "A"와 "A" → true
s[1].equals(s[2]) → "A"와 "A" → true
```

따라서 첫 번째 반복문에서 `OO`가 출력된다.

그 뒤 배열 전체를 출력하면 `AAA`가 출력되므로 최종 결과는 다음과 같다.

```text
OOAAA
```

### 정리

```text
s[0], s[1], s[2]는 배열의 칸이다.
각 칸에는 String 객체 자체가 아니라 String 객체를 가리키는 참조값이 들어 있다.

"A"는 문자열 상수 풀의 객체를 공유할 수 있고,
new String("A")는 내용이 같아도 새 객체를 만든다.

==는 참조값 비교,
equals()는 문자열 내용 비교이다.
```

## 3번 — SQL 3중 중첩 서브쿼리

### 문제 핵심

이 문제는 SQL이 3중으로 중첩되어 있어 복잡해 보이지만, **가장 안쪽 서브쿼리부터 바깥쪽으로 해석**하면 된다.

```sql
SELECT count(*)
FROM employee AS e 
JOIN project AS p 
ON e.project_id = p.project_id
WHERE p.name IN (
    SELECT name 
    FROM project p 
    WHERE p.project_id IN (
        SELECT project_id 
        FROM employee 
        GROUP BY project_id 
        HAVING count(*) < 2
    )
);
```

---

### 1단계: 가장 안쪽 서브쿼리

```sql
SELECT project_id 
FROM employee 
GROUP BY project_id 
HAVING count(*) < 2
```

의미:

```text
employee 테이블을 project_id별로 묶고,
각 프로젝트에 속한 직원 수가 2명 미만인 project_id를 찾는다.
```

즉, **직원이 1명만 배정된 프로젝트 ID**를 찾는 부분이다.

여기서 중요한 것은 `HAVING`이다.

| 구분 | 역할 |
|---|---|
| `WHERE` | 행 하나하나에 조건을 걸 때 사용 |
| `HAVING` | `GROUP BY`로 묶은 그룹 결과에 조건을 걸 때 사용 |

`count(*) < 2`처럼 그룹별 개수에 조건을 걸 때는 `HAVING`을 사용한다.

---

### 2단계: 중간 서브쿼리

```sql
SELECT name 
FROM project p 
WHERE p.project_id IN ( ... )
```

의미:

```text
직원이 1명만 배정된 project_id에 해당하는 프로젝트 이름을 찾는다.
```

즉, **직원이 1명뿐인 프로젝트의 이름(name)**을 구하는 단계다.

---

### 3단계: 바깥 쿼리

```sql
SELECT count(*)
FROM employee AS e 
JOIN project AS p 
ON e.project_id = p.project_id
WHERE p.name IN ( ... )
```

의미:

```text
employee와 project를 project_id 기준으로 조인한 뒤,
프로젝트 이름이 직원 1명뿐인 프로젝트 이름에 해당하는 행의 개수를 센다.
```

결국 전체 쿼리는 다음 의미다.

```text
직원이 1명만 배정된 프로젝트에 속한 employee 행의 개수는 몇 개인가?
```

정답은 다음과 같다.

```text
1
```

### 정리

```text
1. employee를 project_id별로 묶는다.
2. 직원 수가 2명 미만인 project_id를 찾는다.
3. 그 project_id의 project name을 찾는다.
4. employee와 project를 조인한 결과에서 그 name에 해당하는 행 수를 센다.
```

시험에서는 다음 순서로 풀면 된다.

```text
가장 안쪽 SELECT → 중간 SELECT → 바깥 SELECT
```

## 4번 — LRU 페이지 교체 알고리즘

### LRU의 의미

**LRU(Least Recently Used)**는 페이지 교체 알고리즘 중 하나이다.

```text
가장 오랫동안 사용되지 않은 페이지를 교체한다.
```

메모리의 프레임이 꽉 찬 상태에서 새로운 페이지를 넣어야 할 때, 현재 프레임 안의 페이지 중 **가장 오래 사용되지 않은 페이지**를 제거하고 새 페이지를 넣는다.

---

### 페이지와 프레임

| 용어 | 의미 |
|---|---|
| 페이지(Page) | 프로그램을 나눈 조각 |
| 프레임(Frame) | 실제 메모리에서 페이지가 들어가는 칸 |
| 페이지 부재(Page Fault) | 필요한 페이지가 현재 프레임 안에 없는 상황 |

프레임 수가 3개라면, 메모리에는 페이지를 최대 3개까지만 올릴 수 있다.

---

### LRU 풀이 규칙

```text
1. 참조 페이지가 프레임 안에 없으면 페이지 부재가 발생한다.
2. 프레임에 빈칸이 있으면 새 페이지를 넣는다.
3. 프레임이 가득 찼으면 가장 오래 사용되지 않은 페이지를 제거한다.
4. 참조 페이지가 이미 프레임 안에 있으면 페이지 부재는 발생하지 않는다.
5. 이미 있는 페이지를 참조한 경우, 그 페이지는 방금 사용한 것으로 처리한다.
```

특히 5번이 중요하다.

```text
이미 프레임 안에 있는 페이지를 다시 참조하면,
그 페이지는 최근 사용된 것으로 갱신된다.
```

---

### 예시

현재 프레임 상태가 다음과 같다고 하자.

```text
[7, 0, 1]
```

여기서 `0`을 참조하면 `0`은 이미 프레임 안에 있으므로 페이지 부재는 발생하지 않는다.

다만 `0`을 방금 사용했으므로 최근 사용 순서는 갱신된다.

```text
[7, 1, 0]
```

이후 새 페이지가 들어오면 가장 오래 사용되지 않은 `7`이 교체 대상이 된다.

---

### 문제 풀이 결과

페이지 참조 순서:

```text
7 0 1 2 0 3 0 4 2 3 0 3 2 1 2 0 1 7 0 1
```

프레임 수:

```text
3개
```

LRU 방식으로 추적하면 페이지 부재는 총 다음과 같이 발생한다.

```text
7, 0, 1, 2, 3, 4, 2, 3, 0, 1, 0, 7
```

따라서 페이지 부재 횟수는 다음과 같다.

```text
12
```

### 정리

```text
LRU는 프레임이 꽉 찼을 때 가장 오래 사용되지 않은 페이지를 제거한다.
이미 프레임 안에 있는 페이지를 참조하면 페이지 부재는 발생하지 않지만, 최근 사용된 것으로 갱신해야 한다.
```

## 7번 — C언어 `static` 지역변수

### 핵심 개념

C언어에서 함수 안에 선언된 일반 지역변수는 함수가 끝나면 사라진다.

하지만 `static`이 붙은 지역변수는 함수가 끝나도 값이 유지된다.

```c
int func(){
    static int x = 0;
    x += 2;
    return x;
}
```

여기서 `x`는 함수 내부에 선언되었지만, 함수 호출이 끝나도 값이 사라지지 않는다.

---

### 일반 지역변수와 비교

일반 지역변수라면 다음과 같다.

```c
int func(){
    int x = 0;
    x += 2;
    return x;
}
```

이 경우 `func()`가 호출될 때마다 `x`가 새로 만들어지고 0으로 초기화된다.

```text
1회 호출 → 2
2회 호출 → 2
3회 호출 → 2
4회 호출 → 2
```

하지만 static 지역변수는 초기화가 처음 한 번만 실행되고, 이후에는 이전 값을 유지한다.

```text
1회 호출 → x = 0 + 2 = 2
2회 호출 → x = 2 + 2 = 4
3회 호출 → x = 4 + 2 = 6
4회 호출 → x = 6 + 2 = 8
```

---

### 문제 흐름

```c
int func(){
    static int x = 0;
    x += 2;
    return x;
}

int main(){
    int x = 1;
    int sum = 0;
    for(int i=0; i<4; i++) {
        x++;
        sum += func();
    }
    printf("%d", sum);
}
```

`func()`는 총 4번 호출된다.

| 호출 횟수 | 호출 전 static x | `x += 2` 후 | 반환값 |
|---:|---:|---:|---:|
| 1회 | 0 | 2 | 2 |
| 2회 | 2 | 4 | 4 |
| 3회 | 4 | 6 | 6 |
| 4회 | 6 | 8 | 8 |

따라서 `sum`은 다음과 같이 계산된다.

```text
sum = 2 + 4 + 6 + 8 = 20
```

정답:

```text
20
```

---

### `main`의 x와 `func`의 x는 다르다

```c
int main(){
    int x = 1;
}
```

```c
int func(){
    static int x = 0;
}
```

두 변수는 이름은 같지만 서로 다른 변수다.

```text
main의 x → main 함수 안에서만 사용하는 변수
func의 x → func 함수 안에서만 사용하는 static 지역변수
```

따라서 `main` 안의 `x++`는 `func()` 안의 `static int x`에 영향을 주지 않는다.

---

### static 지역변수와 전역변수 차이

`static` 지역변수와 전역변수는 값이 프로그램 실행 중 유지된다는 점에서는 비슷하다.

하지만 접근 범위가 다르다.

| 구분 | 전역변수 | static 지역변수 |
|---|---|---|
| 선언 위치 | 함수 밖 | 함수 안 |
| 값 유지 | 프로그램 끝까지 유지 | 프로그램 끝까지 유지 |
| 접근 가능 범위 | 여러 함수에서 접근 가능 | 선언된 함수 안에서만 접근 가능 |
| 다른 함수에서 직접 수정 | 가능 | 불가능 |
| 특징 | 프로그램 전체에서 공유 | 함수 내부에서만 값 기억 |

즉, static 지역변수는 다음처럼 이해하면 된다.

```text
전역변수처럼 오래 살아남지만,
지역변수처럼 선언된 함수 안에서만 접근 가능한 변수
```

시험 답안용으로는 다음 문장이 가장 정확하다.

```text
static 지역변수는 선언된 함수 내부에서만 접근 가능하지만,
함수 호출이 끝나도 값이 유지되고 초기화는 최초 1회만 수행된다.
```

## 11번 — Java 오버라이딩 메서드와 멤버 변수 접근

### 문제 핵심

```java
class Base {
    int x = 3;
    int getX(){
        return x * 2;
    }
}

class Derivate extends Base {
    int x = 7;
    int getX(){
        return x * 3;
    }
}

public class Main {
    public static void main(String[] args){
        Base a = new Derivate();
        Derivate b = new Derivate();
        System.out.print(a.getX() + a.x + b.getX() + b.x);
    }
}
```

정답:

```text
52
```

---

### 메서드는 실제 객체 타입 기준

```java
Base a = new Derivate();
```

여기서 `a`의 선언 타입과 실제 객체 타입은 다르다.

```text
선언 타입: Base
실제 객체: Derivate
```

메서드가 오버라이딩되어 있으면, 메서드 호출은 **실제 객체 타입 기준**으로 실행된다.

```java
a.getX()
```

`getX()`는 `Derivate`에서 오버라이딩되었으므로, `Base.getX()`가 아니라 `Derivate.getX()`가 실행된다.

```java
class Derivate extends Base {
    int x = 7;
    int getX(){
        return x * 3;
    }
}
```

따라서 `a.getX()`는 다음처럼 계산된다.

```text
a.getX()
→ Derivate.getX() 실행
→ Derivate의 x = 7 사용
→ 7 * 3 = 21
```

---

### `a.getX()` 안의 x는 어느 x인가?

`a.getX()`가 `Derivate.getX()`를 실행하므로, 그 메서드 안의 `x`는 `Derivate`의 `x`를 의미한다.

```java
int getX(){
    return x * 3;
}
```

여기서 `x`는 다음과 같이 볼 수 있다.

```java
return this.x * 3;
```

이때 `this`는 실제 객체인 `Derivate` 객체를 가리킨다.

따라서 `x = 7`이 사용된다.

---

### 멤버 변수는 선언 타입 기준

반면 다음 코드는 다르게 동작한다.

```java
a.x
```

멤버 변수는 오버라이딩되지 않는다. 변수 접근은 **선언 타입 기준**이다.

`a`는 다음처럼 선언되었다.

```java
Base a = new Derivate();
```

따라서 `a.x`는 `Base`의 `x`를 사용한다.

```text
a.x = Base의 x = 3
```

---

### 전체 계산

```java
System.out.print(a.getX() + a.x + b.getX() + b.x);
```

| 식 | 결과 | 이유 |
|---|---:|---|
| `a.getX()` | 21 | 실제 객체가 `Derivate`이므로 `Derivate.getX()` 실행, 내부 x는 `Derivate.x = 7` |
| `a.x` | 3 | 변수 접근은 선언 타입 `Base` 기준 |
| `b.getX()` | 21 | `b`는 선언 타입도 실제 타입도 `Derivate` |
| `b.x` | 7 | `b`는 `Derivate` 타입 변수 |

계산:

```text
21 + 3 + 21 + 7 = 52
```

### 정리

```text
메서드 호출: 실제 객체 타입 기준
멤버 변수 접근: 선언 타입 기준
메서드 안의 x: 그 메서드가 속한 클래스의 x를 우선 사용
```

이 문제에서 `a.getX()`는 `Derivate.getX()`가 실행되므로, 그 안의 `x`는 `Derivate`의 `x = 7`을 사용한다.

## 14번 — UML 클래스 다이어그램 관계

### 핵심 개념

UML 클래스 다이어그램의 관계는 클래스 사이의 연결 방식을 나타낸다. 14번의 핵심은 **연관, 일반화, 의존** 관계를 구분하는 것이다.

---

### UML 클래스 관계 정리표

| 관계 | 의미 | 표기 | 코드 느낌 |
|---|---|---|---|
| 연관 Association | 한 클래스가 다른 클래스를 지속적으로 알고 있음 | 실선 | 멤버 변수, 필드로 가지고 있음 |
| 의존 Dependency | 한 클래스가 다른 클래스를 일시적으로 사용함 | 점선 화살표 | 매개변수, 지역변수, 리턴 타입 |
| 일반화 Generalization | 상속 관계 | 실선 + 빈 삼각형 | `extends` |
| 실체화 Realization | 인터페이스 구현 관계 | 점선 + 빈 삼각형 | `implements` |
| 집합 Aggregation | 약한 전체-부분 관계 | 실선 + 빈 마름모 | 부분이 독립 생존 가능 |
| 합성 Composition | 강한 전체-부분 관계 | 실선 + 검은 마름모 | 전체가 사라지면 부분도 사라짐 |

---

### 연관과 의존의 차이

#### 연관

연관은 한 클래스가 다른 클래스를 **필드로 가지고 지속적으로 참조**하는 관계다.

```java
class Student {
    private School school;
}
```

`Student` 객체는 `school`이라는 필드를 통해 `School` 객체를 계속 알고 있다.

```text
Student ───── School
```

시험식 구분:

```text
A 클래스가 B 클래스를 속성이나 필드로 가지고 있으면 연관
```

---

#### 의존

의존은 한 클래스가 다른 클래스를 **메서드 실행 중에만 잠깐 사용하는 관계**다.

```java
class Student {
    void register(Course course) {
        System.out.println(course.getName());
    }
}
```

`Course`는 `register()` 메서드의 매개변수로 들어와서 잠깐 사용된다.

```text
Student - - - -> Course
```

시험식 구분:

```text
A 클래스가 B 클래스를 매개변수, 지역변수, 리턴 타입 등으로 잠깐 사용하면 의존
```

---

### 연관 vs 의존 한 줄 구분

```text
필드로 가지고 있으면 연관,
메서드 안에서 잠깐 쓰면 의존.
```

---

### 일반화

일반화는 상속 관계다.

```java
class Animal {
}

class Dog extends Animal {
}
```

표기 방향은 자식에서 부모 쪽으로 빈 삼각형이 향한다.

```text
Dog ─────▷ Animal
```

시험식 구분:

```text
is-a 관계이면 일반화 가능성이 높다.
Dog is an Animal.
```

---

### 정리

| 키워드 | 관계 |
|---|---|
| 잠깐 사용, 매개변수, 지역변수 | 의존 |
| 계속 알고 있음, 필드, 멤버 변수 | 연관 |
| 부모-자식, 상속, is-a | 일반화 |
| 인터페이스 구현 | 실체화 |
| 약한 전체-부분, 독립 생존 가능 | 집합 |
| 강한 전체-부분, 생명주기 같음 | 합성 |

---

## 16번 — C언어 이중 포인터와 `*(*arr + i)`

### 문제 핵심

16번은 이중 포인터가 나오지만, 실제 계산은 배열의 각 원소를 수정하는 문제다.

핵심 표현은 다음이다.

```c
*(*arr + i)
```

이 표현은 결국 실제 배열의 `i`번째 값을 의미한다.

---

### 포인터 구조

문제의 구조는 다음과 같다.

```c
int arr[] = {3, 1, 4, 1, 5};
int* p = arr;
int** pp = &p;

func(pp, 5);
```

구조를 그리면 다음과 같다.

```text
pp → p → arr[0]
        → {3, 1, 4, 1, 5}
```

`func(pp, 5)`를 호출했으므로 함수 안의 매개변수 `arr`는 `pp`를 받는다.

```text
함수 안의 arr → p → 실제 배열의 첫 번째 원소
```

---

### `*arr`의 의미

함수 안에서 `arr`는 `int**`이다.

```c
void func(int** arr, int size)
```

따라서 `*arr`는 `arr`가 가리키는 곳으로 한 번 들어간 값이다.

```text
arr  → p
*arr = p
```

`p`는 실제 배열의 첫 번째 원소를 가리킨다.

```text
*arr = p = &arr[0]
```

---

### `*arr + i`의 의미

`*arr`가 실제 배열의 첫 번째 원소 주소라면, `*arr + i`는 배열에서 i칸 이동한 주소다.

```text
*arr + 0 = &arr[0]
*arr + 1 = &arr[1]
*arr + 2 = &arr[2]
*arr + 3 = &arr[3]
*arr + 4 = &arr[4]
```

---

### `*(*arr + i)`의 의미

`*arr + i`는 i번째 원소의 주소다. 그 앞에 `*`를 붙이면 해당 주소에 들어 있는 값을 꺼낸다.

```text
*(*arr + 0) = arr[0]
*(*arr + 1) = arr[1]
*(*arr + 2) = arr[2]
*(*arr + 3) = arr[3]
*(*arr + 4) = arr[4]
```

따라서 다음 표현은 실제 배열의 i번째 원소와 같다.

```c
*(*arr + i)
```

```text
= 실제 배열 arr[i]
```

---

### 함수 내용을 쉽게 바꾸기

원래 코드:

```c
void func(int** arr, int size){
    for(int i=0; i<size; i++){
        *(*arr + i) = (*(*arr+i) + i) % size;
    }
}
```

이 코드는 사실상 다음과 같이 볼 수 있다.

```c
for(int i=0; i<size; i++){
    arr[i] = (arr[i] + i) % size;
}
```

단, 여기서 `arr[i]`는 함수 매개변수 `arr`가 아니라 실제 배열의 i번째 값을 의미한다.

---

### 값 변화

초기 배열:

```text
arr = {3, 1, 4, 1, 5}
```

계산식:

```text
arr[i] = (arr[i] + i) % 5
```

| i | 기존 arr[i] | 계산 | 새 arr[i] |
|---:|---:|---:|---:|
| 0 | 3 | (3+0)%5 | 3 |
| 1 | 1 | (1+1)%5 | 2 |
| 2 | 4 | (4+2)%5 = 6%5 | 1 |
| 3 | 1 | (1+3)%5 | 4 |
| 4 | 5 | (5+4)%5 = 9%5 | 4 |

변경 후 배열:

```text
arr = {3, 2, 1, 4, 4}
```

따라서 `arr[2]`는 다음과 같다.

```text
1
```

---

### 정리

```text
int** pp에서 *pp가 배열 첫 칸을 가리키는 int*라면,
*(*pp + i)는 배열의 i번째 값이다.
```

이 문제는 이중 포인터가 나오지만, 실제 계산은 다음처럼 바꿔서 보면 된다.

```text
arr[i] = (arr[i] + i) % size
```

## 19번 — Java Generic, `value` 할당 과정, 오버로딩

### 문제 핵심

19번은 Java의 **제네릭(Generic)**과 **오버로딩**이 함께 나온 문제다.

핵심은 다음이다.

```text
value 안의 실제 값은 0이지만,
print(value)를 호출할 때는 value의 컴파일 시점 타입을 기준으로 오버로딩 메서드가 선택된다.
```

정답은 다음과 같다.

```text
B0
```

---

### 제네릭이란?

제네릭(Generic)은 클래스나 메서드에서 사용할 자료형을 나중에 정하게 해주는 문법이다.

```java
class Collection<T> {
    T value;
}
```

여기서 `T`는 실제 타입이 아니라 **타입 자리표시자**다.

```text
T = 나중에 정해질 타입
```

예를 들어 다음처럼 사용할 수 있다.

```java
Collection<Integer> a;
Collection<String> b;
```

그러면 다음과 같이 해석된다.

```text
Collection<Integer>에서는 T가 Integer처럼 사용됨
Collection<String>에서는 T가 String처럼 사용됨
```

---

### `<T>`에서 T는 예약어가 아니다

```java
public static class Collection<T>
```

여기서 `T`라는 문자 자체가 제네릭을 의미하는 예약어는 아니다. 중요한 것은 클래스명 뒤에 `<...>`가 붙는 구조다.

```java
class Box<T> { }
class Box<E> { }
class Box<Type> { }
```

모두 가능하다.

다만 관습적으로 다음 이름을 자주 쓴다.

| 이름 | 관습적 의미 |
|---|---|
| `T` | Type |
| `E` | Element |
| `K` | Key |
| `V` | Value |
| `N` | Number |

즉, 자바가 `T`라는 글자를 보고 제네릭으로 이해하는 것이 아니라, **`class 클래스명<타입이름>` 구조를 보고 제네릭 타입 매개변수 선언으로 이해**한다.

---

### `new Collection<>(0)`에서 `<>`의 의미

```java
new Collection<>(0).print();
```

여기서 `<>`는 **다이아몬드 연산자**다.

의미:

```text
타입은 컴파일러가 추론해라.
```

예를 들어 다음처럼 긴 코드를,

```java
Collection<Integer> c = new Collection<Integer>(0);
```

다음처럼 줄일 수 있다.

```java
Collection<Integer> c = new Collection<>(0);
```

오른쪽 `<>` 안을 비워도 왼쪽의 `Collection<Integer>`를 보고 컴파일러가 타입을 추론한다.

문제처럼 왼쪽에 변수 선언이 없는 경우에도,

```java
new Collection<>(0).print();
```

생성자에 들어간 값 `0`을 보고 타입을 추론할 수 있다. `0`은 기본형 `int`지만, 제네릭에는 기본형을 직접 사용할 수 없으므로 `Integer`로 오토박싱된다.

즉, 대략 다음처럼 볼 수 있다.

```java
new Collection<Integer>(0).print();
```

단, `<>`를 비운다는 것은 타입이 없다는 뜻이 아니라, **컴파일러에게 타입 추론을 맡긴다는 뜻**이다.

---

### `value`가 할당되는 과정

문제 구조:

```java
public static class Collection<T>{
    T value;

    public Collection(T t){
        value = t;
    }

    public void print(){
        new Printer().print(value);
    }
}
```

객체 생성:

```java
new Collection<>(0).print();
```

흐름:

```text
1. Collection 객체를 만든다.
2. 생성자 Collection(T t)에 0이 전달된다.
3. t에 0이 들어간다.
4. value = t가 실행된다.
5. value 안에 0이 저장된다.
```

즉, 값 자체만 보면 다음과 같다.

```text
value = 0
```

---

### 오버로딩 메서드 선택

내부 클래스 `Printer`에는 이름이 같은 `print` 메서드가 여러 개 있다.

```java
class Printer{
    void print(Integer a){
        System.out.print("A" + a);
    }
    void print(Object a){
        System.out.print("B" + a);
    }
    void print(Number a){
        System.out.print("C" + a);
    }
}
```

이를 오버로딩이라고 한다.

```text
메서드 이름은 같지만, 매개변수 타입이 다르다.
```

주의할 점:

```text
오버로딩은 실행 중 실제 값이 아니라 컴파일 시점의 타입을 기준으로 선택된다.
```

`value` 안의 실제 값은 `0`이지만, `value`의 선언 타입은 다음과 같다.

```java
T value;
```

제네릭의 `T`는 오버로딩 판단에서 구체적으로 `Integer`라고 확정되지 않는다. 따라서 `print(value)`는 `print(Object)` 쪽으로 선택된다.

```java
void print(Object a){
    System.out.print("B" + a);
}
```

출력:

```text
B0
```

---

### `Integer`, `Number`, `Object` 관계

```text
Object
  └─ Number
       └─ Integer
```

만약 변수 타입이 각각 다음과 같았다면 결과가 달라진다.

| 선언 타입 | 호출 메서드 | 출력 |
|---|---|---|
| `Integer value = 0;` | `print(Integer)` | `A0` |
| `Number value = 0;` | `print(Number)` | `C0` |
| `Object value = 0;` | `print(Object)` | `B0` |
| `T value;` | 오버로딩 판단에서 Object 쪽 | `B0` |

---

### 정리

```text
Generic은 타입을 나중에 정하게 해주는 문법이다.
<T>에서 T는 타입 매개변수 이름이며, T 자체가 예약어는 아니다.
new Collection<>(0)의 <>는 다이아몬드 연산자로, 타입 추론을 의미한다.

생성자에서 0이 value에 저장된다.
하지만 print(value)에서 어떤 print가 호출될지는 value 안의 실제 값이 아니라 value의 컴파일 시점 타입으로 결정된다.
제네릭 T 타입 변수는 오버로딩 판단에서 Object 쪽이 선택되어 B0이 출력된다.
```

## 20번 — 네트워크 형태/구성 방식 용어 정리

### 문제 핵심

20번의 정답은 **Ad-hoc Network**다.

핵심 설명:

```text
중앙 관리나 고정 인프라 없이 임시로 구성되는 네트워크
```

---

### 보기 용어 정리

| 용어 | 뜻 | 핵심 키워드 | 예시 |
|---|---|---|---|
| Infrastructure Network | AP, 기지국, 라우터 같은 고정 인프라를 중심으로 구성되는 네트워크 | 고정 장비, 중앙 장치, AP | 공유기 기반 Wi-Fi |
| Firmware Network | 일반적인 표준 네트워크 분류 용어로는 잘 쓰이지 않음 | 펌웨어는 장치 내장 소프트웨어 | 라우터 펌웨어, IoT 펌웨어 |
| Peer-to-Peer Network, P2P | 중앙 서버 없이 동등한 노드끼리 직접 자원을 주고받는 네트워크 | 동등한 노드, 직접 공유 | 파일 공유, 블록체인 노드 |
| Ad-hoc Network | 고정 인프라 없이 노드들이 임시로 직접 연결해 구성하는 네트워크 | 임시 구성, 무선, 중앙 장치 없음 | 재난 구조, 군사 작전, 긴급 회의 |
| Mesh Network | 여러 노드가 그물망처럼 연결되어 여러 경로로 통신 가능한 네트워크 | 그물망, 다중 경로, 우회 가능 | Wi-Fi Mesh, 군 통신망 |
| Sensor Network | 여러 센서 노드가 데이터를 수집하고 전달하는 네트워크 | 센서, 측정, 수집 | 온도·습도·진동 센서망 |
| Virtual Private Network, VPN | 공용망 위에 암호화된 사설 통신 터널을 만드는 기술 | 사설망 확장, 암호화, 터널링 | 회사 VPN, IPsec/SSL VPN |

---

### 헷갈리는 용어 비교

| 구분 | Ad-hoc Network | Mesh Network | P2P Network |
|---|---|---|---|
| 중심 개념 | 임시 네트워크 구성 | 그물망 구조와 다중 경로 | 동등한 노드 간 직접 공유 |
| 고정 인프라 | 없음 | 없을 수도 있고 있을 수도 있음 | 중앙 서버 없이 가능 |
| 시험 키워드 | 긴급, 임시, 군사, 재난, 무선 | 다중 경로, 우회, 그물망 | peer, 동등, 파일 공유 |
| 이번 문제 정답 여부 | 정답 | 오답 가능성 높은 보기 | 오답 가능성 있는 보기 |

### 정리

```text
긴급 구조, 긴급 회의, 군사적 상황, 고정 인프라 없이 임시 구성
→ Ad-hoc Network
```
