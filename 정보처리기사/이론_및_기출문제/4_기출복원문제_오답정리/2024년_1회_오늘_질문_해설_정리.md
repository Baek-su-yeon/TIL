# 6번 - 정규형 판별 (3NF vs BCNF)

| 고객아이디 | 강좌명 | 강사번호 |
| --- | --- | --- |
| apple | 영어회화 | P001 |
| banana | 기초토익 | P002 |
| carrot | 영어회화 | P001 |
| carrot | 기초토익 | P004 |
| orange | 영어회화 | P003 |
| orange | 기초토익 | P004 |

**정답: 제3정규형**

> ⚠️ **해설 파일(`2024년_1회_해설.md`)의 6번 해설은 부실하다.**
> "이행적 종속이 없으므로 제3정규형"이라고만 되어 있는데, 답은 맞지만 **이유가 틀렸다.**
> 이 문제의 핵심은 *"왜 BCNF가 아닌가"* 이다. 그 해설대로 외우면 다음에 또 틀린다.

## 핵심

이 문제가 묻는 진짜 질문은 이것이다.

> **"3NF까지는 되는데, 왜 BCNF는 안 되는가?"**

시나공/이기적에 나오는 **"3NF지만 BCNF가 아닌" 대표 예제**다.

### 함수 종속 찾기

**① (고객아이디, 강좌명) → 강사번호**

carrot이 영어회화를 들으면 P001, 기초토익을 들으면 P004. 둘 다 있어야 강사가 정해진다.
→ **기본키 = (고객아이디, 강좌명)**

**② 강사번호 → 강좌명** ← **숨은 범인**

강사번호 열을 **세로로** 훑는다:

| 강사번호 | 강좌명 |
| --- | --- |
| P001 | 영어회화, 영어회화 |
| P002 | 기초토익 |
| P003 | 영어회화 |
| P004 | 기초토익, 기초토익 |

**P001은 어디 나오든 무조건 영어회화. P004는 무조건 기초토익.** 모순되는 행이 하나도 없다.
→ **"강사 한 명은 한 과목만 가르친다"**는 규칙이 표에 숨어있다.

**반대는 성립하지 않는다.** `강좌명 → 강사번호`? 영어회화가 P001이기도 P003이기도 하다. ❌

### 판정

**3NF는 통과** ⭕
```
강사번호 → 강좌명
  강사번호는 후보키인가?  → 아님 (P001만 보고 apple/carrot 구분 불가) → ① 탈락
  강좌명은 기본속성인가?  → O (기본키 (고객아이디, 강좌명)의 일부) → ② 통과!
```

**BCNF는 위반** ❌
```
BCNF에는 ②번 탈출구가 없다.
강사번호가 후보키가 아니므로 → 위반
```

**→ 만족하는 최고 단계가 3NF. 정답은 제3정규형.**

### 3NF와 BCNF의 유일한 차이

```
3NF  : 모든 X → A 에서
       ① X가 후보키다              ← 또는
       ② A가 기본속성이다           ← 이 탈출구가 있음!

BCNF : 모든 X → A 에서
       ① X가 후보키다              ← 이것뿐. 탈출구 없음
```

> **BCNF = 모든 결정자는 후보키여야 한다.** 이 한 줄이 전부다.

### 해결

```
수강(고객아이디, 강사번호)
강사(강사번호, 강좌명)        ← 결정자를 아예 키로 만들어 독립시킴
```

## 다음에 푸는 기준

```
표 주고 "몇 정규형이냐" 물으면서
키가 아닌 열이 다른 열을 결정하는 구조가 보이면
→ 답은 3NF (BCNF 함정 문제)

찾는 법: 열을 "세로로" 훑어서
        같은 값일 때 다른 열도 항상 같은지 본다
        ← 6번의 승부처. 강사번호 열을 세로로 안 보면 절대 못 찾음
```

# 6번 확장 - 정규화 완전 정리

> 정규화 개념 전체를 모른다고 하여 1NF~5NF를 처음부터 정리한다.

## 0단계 - 함수 종속 (모든 것의 기초)

### X → Y 의 뜻

> **"X가 정해지면 Y가 딱 하나로 정해진다"**

```
학번 → 이름     ⭕ 학번 1001이면 이름은 "백수연" 하나
이름 → 학번     ❌ 동명이인이 있으면 학번이 안 정해짐
```

X를 **결정자(Determinant)**라 부른다.

### 표에서 찾는 법

> **X 열의 같은 값을 가진 행들을 모아서, Y 열도 전부 같은지 본다. 하나라도 다르면 종속 아님.**

**함수 종속은 반례가 단 하나만 있어도 성립하지 않는다.**

---

## 키(Key) 개념 - 여기가 뿌리다

> 기본키/후보키를 헷갈리면 정규화 판별이 아예 안 된다.

### 관계도

```
슈퍼키  ⊃  후보키  ⊃  기본키
                  └→ 나머지 = 대체키
```

| 키 | 조건 | 설명 |
| --- | --- | --- |
| **슈퍼키** | 유일성 | 행 구분만 되면 됨. **군더더기 있어도 OK** |
| **후보키** | 유일성 + **최소성** + NOT NULL | 군더더기 없어야 함. **여러 개 가능** |
| **기본키** | 후보키 중 **선택** | ← **계산이 아니라 사람이 고르는 것** |
| **대체키** | 기본키가 안 된 후보키 | |
| **외래키** | 다른 릴레이션의 기본키 참조 | **NULL 허용됨** |

### 슈퍼키 vs 후보키 - 최소성이 차이

학생(학번, 이름, 학과)에서 학번만 유일하다고 하면:

| 조합 | 유일성 | 최소성 | 판정 |
| --- | --- | --- | --- |
| (학번) | O | O | **후보키** |
| (학번, 이름) | O | **X** | 슈퍼키 |
| (학번, 이름, 학과) | O | **X** | 슈퍼키 |
| (이름) | X | - | 아무것도 아님 |

`(학번, 이름)`은 학번만으로 이미 구분되므로 **이름이 군더더기**다. 뺄 수 있는 게 있으면 최소성 위반.

> **모든 후보키는 슈퍼키다. 하지만 모든 슈퍼키가 후보키는 아니다.**

### 후보키 vs 기본키 - 여기가 헷갈리는 지점

학생(학번, **주민등록번호**, 이름, 학과)

```
후보키: 학번, 주민등록번호       ← 둘 다 자격이 있음 (여러 개 가능!)
기본키: 학번                    ← 그중 설계자가 "고른" 하나
대체키: 주민등록번호             ← 뽑히지 못한 나머지
```

> **기본키는 논리적으로 계산되어 나오는 게 아니라, 사람이 선택하는 것이다.**

**기본키 선택 기준**
```
① NOT NULL   — 반드시 값이 있어야 함 (개체 무결성)
② 유일성      — 중복 불가
③ 최소성      — 군더더기 없이
④ 불변성      — 잘 안 바뀌는 값 (휴대폰번호 ✗)
⑤ 단순성      — 짧고 간단하게
```

### 기본속성 vs 비기본속성 - 3NF/BCNF를 가르는 칼

```
기본속성(Prime)     = 후보키에 "포함된" 속성
비기본속성(Non-prime) = 후보키에 "안 들어간" 속성
```

기본키가 (학번, 과목코드)일 때:

| 속성 | 분류 |
| --- | --- |
| 학번, 과목코드 | **기본속성** |
| 성적, 학생이름 | 비기본속성 |

**3NF의 ②번 탈출구가 "A가 기본속성이냐"를 묻는다.** 이 구분을 못 하면 3NF와 BCNF를 절대 구분할 수 없다.

### 후보키 찾는 법 - 폐포(Closure)

> **어떤 속성에서 출발해 화살표를 계속 따라갔을 때 전체 속성이 다 나오면, 그게 후보키다.**

```
R(사번, 사원명, 부서코드, 부서명)
FD: 사번 → 사원명,  사번 → 부서코드,  부서코드 → 부서명

사번⁺ = {사번}
      → {사번, 사원명}
      → {사번, 사원명, 부서코드}
      → {사번, 사원명, 부서코드, 부서명}   = 전체!
∴ 사번이 후보키
```

**요령:** 한 번도 **화살표의 오른쪽에 안 나오는 속성**은 반드시 후보키에 포함된다. (아무도 그 값을 만들어주지 못하므로)

### 무결성 3종 (시험 단골)

| 무결성 | 대상 | 규칙 |
| --- | --- | --- |
| **개체 무결성** | 기본키 | NULL 불가, 중복 불가 |
| **참조 무결성** | 외래키 | 참조 대상에 있는 값이거나 **NULL** |
| **도메인 무결성** | 일반 속성 | 정해진 자료형·범위 준수 |

> **외래키는 NULL이 허용된다.** 기본키(NULL 불가)와 헷갈리지 말 것. 자주 나오는 함정.

## 이상현상 3종 - 정규화를 하는 이유

| 이상 | 증상 |
| --- | --- |
| **삽입 이상** | 넣고 싶은 것만 못 넣음. 쓸데없는 값을 억지로 채워야 함 |
| **삭제 이상** | 하나 지웠는데 **남겨야 할 정보까지** 딸려 사라짐 |
| **갱신 이상** | 같은 값이 여러 행에 흩어져, 일부만 고치면 **데이터 불일치** |

## 1NF - 원자값

> **표의 한 칸에 값이 하나씩만 들어있어야 한다**

### 위반 ①: 한 칸에 값 여러 개

| 학번 | 이름 | 수강과목 |
| --- | --- | --- |
| 1001 | 백수연 | **데이터베이스, 네트워크** |

`WHERE 수강과목 = '네트워크'` 로 검색하면 **안 잡힌다.** 그 칸의 값은 `'네트워크'`가 아니라 `'데이터베이스, 네트워크'`라는 통짜 문자열이다.

### 위반 ②: 반복 그룹

| 학번 | 이름 | 수강과목1 | 수강과목2 | 수강과목3 |
| --- | --- | --- | --- | --- |
| 1001 | 백수연 | 데이터베이스 | 네트워크 | |

**칸마다 값은 하나씩이라 괜찮아 보이는 게 함정이다.** 이것도 1NF 위반이다.

- 검색이 지옥: `WHERE 과목1='X' OR 과목2='X' OR 과목3='X'`
- **4과목 듣는 학생이 나타나면 테이블 구조를 뜯어고쳐야 한다.** 데이터가 늘었다고 스키마를 바꾸는 건 설계 실패.
- NULL이 잔뜩 생김

### 해결 - 행으로 펼친다

| 학번 | 이름 | 수강과목 |
| --- | --- | --- |
| 1001 | 백수연 | 데이터베이스 |
| 1001 | 백수연 | 네트워크 |

이제 10과목이든 **행만 늘리면 된다. 구조는 그대로.**

## 2NF - 부분 함수 종속 제거

> **기본키가 복합키일 때, 키의 "일부분"만으로 정해지는 속성이 있는 것**

반대말이 **완전 함수 종속** — 키 전부가 있어야 정해지는 것.

### 예제 / 기본키 = (학번, 과목코드)

| 학번 | 과목코드 | 성적 | 학생이름 |
| --- | --- | --- | --- |
| 1001 | DB | A | 백수연 |
| 1001 | NET | B | 백수연 |
| 1002 | DB | C | 김철수 |

**키를 쪼개서 하나씩 물어본다:**

```
성적을 알려면?
  "1001의 DB 성적?" → A      ✅ 둘 다 필요
  "1001의 성적?"     → A? B?  ❌
  "DB 성적?"         → A? C?  ❌
  → 완전 함수 종속 ⭕ 정상

학생이름을 알려면?
  "1001의 이름?" → 백수연      ✅ 과목코드가 필요 없다!
  → 부분 함수 종속 ❌
```

### 무엇이 문제인가

**"백수연"이 중복 저장된다.** 과목 10개 들으면 10번.

- **갱신 이상**: 이름 바꾸려면 10개 행을 다 고쳐야 함. 9개만 고치면 모순
- **삽입 이상**: 수강신청 안 한 신입생은 **등록조차 못 함** (과목코드가 기본키 일부라 NULL 불가)
- **삭제 이상**: 수강 취소로 행이 다 지워지면 **학생 존재 자체가 증발**

### 해결

```
수강(학번, 과목코드, 성적)     ← 키 전체가 필요한 것만
학생(학번, 학생이름)           ← 부분종속된 놈을 분리
```

### 🔑 결정적 요령

> **기본키가 1개짜리면 부분종속은 애초에 불가능하다. (쪼갤 "일부"가 없으므로)**
> **→ 단일키 테이블은 1NF면 자동으로 2NF다.**

**"2NF 위반인가?" 물으면 → 기본키가 복합키인지부터 확인.**

## 3NF - 이행 함수 종속 제거

> **A → B → C 로 징검다리를 건너서 A → C 가 간접 성립하는 것** (단, B가 키가 아닐 때)

### 예제 / 기본키 = 학번 (단일키)

| 학번 | 이름 | 학과코드 | 학과명 |
| --- | --- | --- | --- |
| 1001 | 백수연 | D01 | 컴퓨터공학과 |
| 1002 | 김철수 | D01 | 컴퓨터공학과 |
| 1003 | 이영희 | D02 | 전자공학과 |

**이 표는 2NF는 만족한다** (단일키라 부분종속 불가). **그런데도 "컴퓨터공학과"가 중복된다.**
→ **이게 3NF가 따로 필요한 이유다.**

```
학번 → 학과코드      (A → B)   1001이면 D01
학과코드 → 학과명    (B → C)   D01이면 컴퓨터공학과
─────────────────────────────
∴ 학번 → 학과명      (A → C)   이행종속 ❌
```

**중간에 낀 B(학과코드)가 키가 아니라는 게 포인트.** 키도 아닌 놈이 남의 다리 노릇을 한다.

### 무엇이 문제인가

- **갱신 이상**: 학과명 개편 시 소속 학생 300명 행을 전부 수정
- **삽입 이상**: **신설 학과를 학생 없이 등록 못 함**
- **삭제 이상**: 이영희 자퇴 → **"D02 = 전자공학과" 정보가 통째로 사라짐**

### 해결

```
학생(학번, 이름, 학과코드)     ← 학과코드는 외래키로 남김
학과(학과코드, 학과명)         ← 징검다리(B→C)를 독립 테이블로
```

### 2NF vs 3NF 구분

| | 2NF (부분종속) | 3NF (이행종속) |
| --- | --- | --- |
| **모양** | (A,B) → C 인데 **A → C** | **A → B → C** |
| **한 마디** | 키를 **덜 쓰고도** 알아냄 | **남의 다리 건너서** 알아냄 |
| **언제** | **복합키**일 때만 | **단일키에서도** 발생 |

> ⚠️ **복합키에서도 이행종속은 생긴다.** 복합키를 봤다고 2NF만 보고 끝내면 안 된다.
> (예: (학번,과목) → 점수 → 학점)

---

## BCNF - 모든 결정자가 후보키

**강한 3NF**라고도 한다. → **6번 문제의 정답 근거. 위 6번 항목 참조.**

```
결정자인데 후보키가 아닌 놈이 있으면 → BCNF 위반 → 답은 3NF
```

---

## 4NF - 다치 종속 제거

여기서부터 **함수 종속의 세계를 떠난다.** 화살표가 두 개(**↠**)로 바뀐다.

> **A ↠ B : A 하나에 B가 여러 개 대응되는데,**
> **그 B 목록이 다른 속성과 아무 상관없이 독립적으로 정해지는 것**

핵심 단어는 **"독립"**.

### 예제

| 교수 | 과목 | 취미 |
| --- | --- | --- |
| 김교수 | DB | 등산 |
| 김교수 | DB | 낚시 |
| 김교수 | 네트워크 | 등산 |
| 김교수 | 네트워크 | 낚시 |

```
교수 ↠ 과목     김교수 → {DB, 네트워크}
교수 ↠ 취미     김교수 → {등산, 낚시}
```

**과목과 취미는 서로 완전히 무관하다.** DB를 가르쳐서 등산하는 게 아니다.
그런데 한 테이블에 있으니 **모든 조합(2×2=4행)**을 다 만들어야 한다.

### 왜 BCNF로는 못 잡나 (4NF의 존재 이유)

이 표에는 **함수 종속이 하나도 없다.**

```
교수 → 과목?  ❌ 김교수의 과목이 DB인지 네트워크인지 하나로 안 정해짐
교수 → 취미?  ❌ 마찬가지
```

**화살표 하나짜리(→)가 없다 = 결정자가 없다 = BCNF가 위반할 게 없어서 그냥 통과.**
(기본키는 세 속성 전부)

**그런데도 중복이 심하다.** → **함수 종속만으로는 못 잡는 중복이 존재한다.**

### 무엇이 문제인가

과목 하나 추가 → 취미 개수만큼 행 추가. **곱셈으로 폭발한다.**

```
과목 3개 × 취미 4개 = 12행 😱
```

### 해결 - 독립적인 두 덩어리를 분리

```
담당(교수, 과목)
취미(교수, 취미)
```

```
분리 전:  과목 3개 × 취미 4개 = 12행
분리 후:  과목 3행 + 취미 4행 =  7행
```

> **곱셈이 덧셈으로 바뀐다. 이게 4NF의 효과.**

---

## 5NF - 조인 종속 제거

**PJNF**(Project-Join Normal Form)라고도 한다.

> **실기 출제 거의 없음. 용어 매칭만 하면 된다.**

```
"조인 종속" → 5NF
```

### 개념

> **테이블을 여러 개로 쪼갰다가 다시 조인하면 원래대로 복원되는 성질**
> **5NF = 더 이상 무손실 분해가 불가능한 상태**

### 4NF와의 차이 - 이게 전부

| | 분해 |
| --- | --- |
| **4NF** | **2개**로 쪼개면 해결 |
| **5NF** | **2개로는 안 되고, 3개로 쪼개야** 복원됨 |

### 예제

판매(사원, 제품, 거래처)
규칙: *"사원이 그 제품을 취급 & 그 거래처를 담당 & 그 거래처가 그 제품을 취급하면 → 그 사원은 그 거래처에 그 제품을 판다"*

| 사원 | 제품 | 거래처 |
| --- | --- | --- |
| 김 | 노트북 | A상사 |
| 김 | 마우스 | B상사 |
| 이 | 노트북 | B상사 |
| **김** | **노트북** | **B상사** | ← 위 3행에서 규칙에 의해 유도되는 행 = 불필요한 중복 |

3개로 분해하면 셋을 다 조인해야 복원된다. 둘만 조인하면 **가짜 행(spurious tuple)**이 생긴다.

---

## 정규화 총정리

| 정규형 | 제거 대상 | 신호 |
| --- | --- | --- |
| **1NF** | 반복 그룹 | 한 칸에 쉼표 / 열 이름에 숫자(상품1,2,3) |
| **2NF** | **부분** 함수 종속 | 복합키인데 **키 한쪽만 봐도** 정해지는 속성 |
| **3NF** | **이행** 함수 종속 | **A → B → C** 징검다리 (B가 키 아님) |
| **BCNF** | 후보키 아닌 **결정자** | **키 아닌 열이 키의 일부인 열을 결정** |
| **4NF** | **다치** 종속 | 함수종속 없음 + 무관한 목록 2개의 **모든 조합** |
| **5NF** | **조인** 종속 | 더 이상 분해 불가 |

### 🎯 두문자 암기

```
원  -  부  -  이  -  결  -  다  -  조
원자값 부분종속 이행종속 결정자 다치종속 조인종속
1NF     2NF      3NF     BCNF    4NF      5NF
```

### 실전 판정 순서도

```
1. 한 칸에 값이 여러 개인가?          → YES: 비정규형
2. 기본키가 복합키인가?
   NO  → 부분종속 불가. 2NF 자동 통과. 3번으로
   YES → 키 일부만으로 정해지는 속성? → YES: 1NF (2NF 위반)
3. A → B → C 징검다리 있나?           → YES: 2NF (3NF 위반)
4. 결정자 중 후보키가 아닌 놈 있나?    → YES: 3NF (BCNF 위반)
5. 독립적 다중값 2개가 조합 폭발?      → YES: BCNF (4NF 위반)
```

### ⚠️ 부분종속 vs BCNF위반 - 결정적 구분

둘 다 "키 아닌 화살표"라 헷갈린다. **화살표 오른쪽이 기본키의 일부냐 아니냐**로 갈린다.

```
과목코드 → 과목명    과목코드=키의 일부,  과목명=비기본속성   → 부분종속 → 답 1NF
강사번호 → 강좌명    강사번호=키 아님,    강좌명=기본속성!    → BCNF위반 → 답 3NF
```

```
부분종속 (→ 답 1NF)
   (A, B) → C  인데  A → C
   왼쪽이 키의 일부,  오른쪽이 비기본속성

이행종속 (→ 답 2NF)
   A → B → C
   B가 키가 아님,    C가 비기본속성

BCNF위반 (→ 답 3NF)
   (A, B) → C  인데  C → B
   왼쪽이 후보키 아님,  오른쪽이 기본속성!  ← 여기가 차이
```

### 마지막 - 반정규화

> **정규화는 단계가 높을수록 중복이 적다. 하지만 무조건 높은 게 정답은 아니다.**
> 테이블이 잘게 쪼개질수록 **조인이 많아져 조회 성능이 떨어진다.**
> 그래서 실무에선 일부러 되돌린다 → **반정규화(De-normalization)**. 이것도 시험에 나온다.

# 7번 - OSPF (라우팅 프로토콜)

**정답: OSPF (Open Shortest Path First)**

## 핵심

**이름이 곧 답이다.**

> **O**pen **S**hortest **P**ath **F**irst = "**최단 경로 우선**"
> Shortest Path First → 최단 경로 → **다익스트라**

이름에 알고리즘이 박혀있다. 이거 하나만 연결되면 3번 지문은 자동이다.

## 라우팅 프로토콜 전체 지도

**IGP vs EGP부터 가른다.**

```
IGP (내부 게이트웨이) — 자율 시스템(AS) "안"
├─ RIP        거리 벡터, 홉 카운트, 최대 15홉, 소규모
├─ OSPF       링크 상태, 다익스트라, 대규모
└─ IGRP/EIGRP 시스코 전용

EGP (외부 게이트웨이) — 자율 시스템 "사이"
└─ BGP        경로 벡터, 인터넷 백본
```

**7번 2번 지문이 "단일 자율 시스템 내부" = IGP.** 여기서 이미 BGP는 탈락하고 RIP/OSPF로 좁혀진다. 그다음 "다익스트라"로 OSPF 확정.

## RIP vs OSPF 비교표

| | RIP | OSPF |
| --- | --- | --- |
| 방식 | **거리 벡터** | **링크 상태** |
| 알고리즘 | 벨만-포드 | **다익스트라** |
| 기준(metric) | **홉 수** | 비용(대역폭) |
| 최대 홉 | **15홉** | 제한 없음 |
| 규모 | 소규모 | 대규모 |
| 갱신 | 30초마다 **전체** 전송 | **변화 있을 때만** |

## 다음에 푸는 기준

```
"링크 상태" / "Dijkstra"        → OSPF
"거리 벡터" / "홉 카운트" / "15" → RIP
"AS 간" / "인터넷 백본"          → BGP
"단일 자율 시스템 내부"           → IGP (RIP or OSPF)

RIP  = 홉만 세는 단순무식. 15홉 넘으면 못 감 → 작은 동네용
OSPF = 네트워크 전체 지도로 최단경로 계산    → 큰 도시용
BGP  = 나라와 나라 사이
```

# 8번 - 조인의 종류 (관계대수)

**정답: (1) 세타 조인 / (2) 동등 조인 / (3) 자연 조인**

## 질문: 이게 관계대수의 조인인가?

> **완전히 관계대수 문제다.** σ(셀렉트)·π(프로젝트)와 같은 식구다.

## 관계대수 연산자 지도

```
관계대수
│
├─ 순수 관계 연산자  (관계 DB만의 연산)
│   ├─ SELECT   σ (시그마)     행 고르기
│   ├─ PROJECT  π (파이)       열 고르기
│   ├─ JOIN     ⋈ (나비넥타이)  두 릴레이션 연결   ← 8번이 여기!
│   └─ DIVISION ÷              나누기
│
└─ 일반 집합 연산자  (수학 집합에서 빌려옴)
    ├─ 합집합    ∪
    ├─ 교집합    ∩
    ├─ 차집합    −
    └─ 카티션 프로덕트  ×
```

**8번의 세타/동등/자연 조인은 저 `JOIN ⋈` 하나를 세분한 것이다.**

## 핵심 - 조인은 "카티션 프로덕트 + 셀렉트"다

**조인은 새로운 연산이 아니라 기존 연산 두 개의 조합이다.**

```
R ⋈θ S  =  σθ (R × S)
세타조인      셀렉트(카티션 프로덕트)
```

1. `R × S` — 두 테이블의 **모든 조합**을 다 만든다 (3행 × 4행 = 12행)
2. `σθ` — 그중 **조건 θ를 만족하는 행만** 골라낸다

즉 **"일단 다 곱해놓고 조건으로 거른다"**가 조인의 정체다. θ(세타)는 그냥 **조건**을 뜻하는 기호.

## 세 가지 구분

| 조인 | 기호 | 조건 | 설명 |
| --- | --- | --- | --- |
| **세타 조인** | R ⋈θ S | `=, ≠, >, <, ≥, ≤` | **모든 비교 연산자** 사용 가능. 가장 넓은 개념 |
| **동등 조인** | R ⋈(=) S | `=` 만 | 세타 조인의 **특수 케이스** |
| **자연 조인** | R ⋈ S | `=` + 중복열 제거 | 동등 조인 결과에서 **중복 속성을 π로 제거** |

### 자연 조인이 왜 필요한가

동등 조인을 하면 결과에 이런 일이 생긴다:

| 학번 | 이름 | **R.학과코드** | **S.학과코드** | 학과명 |
| --- | --- | --- | --- | --- |
| 1001 | 김민수 | D01 | D01 | 컴퓨터공학과 |

**학과코드가 두 번 나온다.** `=`로 이었으니 값이 당연히 같다. 하나는 군더더기.

자연 조인은 그걸 하나로 합친다:

| 학번 | 이름 | 학과코드 | 학과명 |
| --- | --- | --- | --- |
| 1001 | 김민수 | D01 | 컴퓨터공학과 |

**자연 조인 기호엔 θ가 안 붙는다** (`⋈` 만). 조건을 안 써도 **같은 이름의 열끼리 자동으로** 이어주기 때문.

## 포함 관계 - 8번의 전부

```
세타 조인  ⊃  동등 조인  ⊃  자연 조인
(모든 비교)   (=만)        (=만 + 중복열 제거)

     넓다 ────────────────────→ 좁다
```

**8번 지문이 정확히 이 계단을 내려간다:**
```
(1) "속성 값을 비교하여 조건을 만족"   → 비교 일반    → 세타
(2) "정확하게 '=' 등호로 일치"        → =로 좁힘     → 동등
(3) "(2)에서 중복된 속성을 제거"      → 중복열 제거   → 자연
```

> **"(3)이 ((2)) 조인에서"라고 문제에 써있는 게 힌트다.** 자연 조인이 동등 조인의 하위라는 걸 문제가 대놓고 알려준 것.

## SQL과의 연결

```sql
-- 세타 조인
SELECT * FROM R, S WHERE R.나이 > S.나이;

-- 동등 조인
SELECT * FROM R, S WHERE R.학과코드 = S.학과코드;
SELECT * FROM R INNER JOIN S ON R.학과코드 = S.학과코드;

-- 자연 조인
SELECT * FROM R NATURAL JOIN S;   -- 조건 안 씀! 같은 이름 열끼리 자동
```

**`NATURAL JOIN`에 `ON`이 없는 이유**: 조건이 "같은 이름 열끼리 =" 로 고정되어 있으므로.

## 내부 조인 vs 외부 조인

```
내부 조인(INNER): 세타, 동등, 자연     ← 8번 범위. 조건 만족하는 것만 나옴
외부 조인(OUTER): LEFT, RIGHT, FULL    ← 관계대수의 "확장 연산"
                                         조건 안 맞아도 한쪽을 살려둠
```

## 다음에 푸는 기준

```
세타 = 모든 비교 (=, >, <, ≠ ...)
동등 = '=' 만
자연 = '=' + 중복 속성 제거

세타 ⊃ 동등 ⊃ 자연

조인 = 카티션 프로덕트(×) + 셀렉트(σ)
→ 따로 외울 게 아니라 σ·π·× 의 조합으로 이해할 것
```

# 9번 - LRU / LFU

```
페이지 참조 순서: 1, 2, 3, 1, 2, 4, 1, 2, 5, 7   (프레임 3개)
```

**정답: LRU 6회 / LFU 6회**

## 질문: LFU는 LRU랑 거의 비슷한 거 아닌가?

> **완전히 다른 알고리즘이다. 이 문제에서 우연히 답이 같았을 뿐이다.**
> 감으로 풀려도 맞은 이유가 그것이고, 그게 위험하다.

## 핵심 - 기준이 근본적으로 다르다

```
LRU  =  Least Recently Used   →  "언제" 썼나   (시간)
LFU  =  Least Frequently Used →  "몇 번" 썼나  (횟수)
```

**LRU는 시계를 보고, LFU는 계산기를 본다.**

## 둘이 정반대로 행동하는 예

프레임 3개, 참조열: `1, 1, 1, 1, 1, 2, 3, 4, 2, 3, 4`
(1을 5번 몰아 쓰고, 그 뒤로 1은 다시 안 나옴)

### LRU

```
1×5회:  [1]                       부재 1회
2:      [1,2]                     부재
3:      [1,2,3]                   부재
4:      가장 오래 안 쓴 게 뭐지?
        1 → 5번째에 마지막 사용  ← 제일 오래됨!
        1을 버림 → [2,3,4]        부재
2:      hit ✅    3: hit ✅    4: hit ✅
```
**부재 4회**

### LFU

```
1×5회:  [1] count=5
2:      [1,2] c2=1                부재
3:      [1,2,3] c3=1              부재
4:      횟수 최소가 뭐지?
        1 → 5회  ← 절대 안 버림!
        2를 버림 → [1,3,4]        부재
2:      부재! → 3 버림 → [1,4,2]
3:      부재! → 4 버림 → [1,2,3]
4:      부재! → 2 버림 → [1,3,4]
```
**부재 7회**

### 결과

| | 부재 |
| --- | --- |
| LRU | **4회** |
| LFU | **7회** |

> **LFU의 유명한 약점:** 초반에 반짝 많이 쓰인 페이지가 **영원히 안 나간다.**
> 다시는 안 쓸 1을 과거 기록(5회) 때문에 끝까지 붙들고 있어, 정작 필요한 2·3·4가 서로 밀어내며 계속 부재가 터진다.
> LRU는 시간을 보니 미련 없이 버린다.

## 그럼 왜 9번에선 6/6으로 같았나

```
참조열: 1, 2, 3, 1, 2, 4, 1, 2, 5, 7
```

**이 참조열은 1과 2가 "최근에도 썼고 + 많이도 썼다".** 두 기준에서 **동시에 1등**이다.
반대로 3, 4, 5는 **딱 한 번 쓰이고 방치**된다.

- LRU 눈에도 → 제일 오래된 놈
- LFU 눈에도 → 제일 적게 쓴 놈

**희생자가 매번 같은 놈으로 찍힌다.** 그래서 결과가 같아진 것. **우연이다.**

### 희생자 비교

| 시점 | LRU가 버리는 것 | LFU가 버리는 것 |
| --- | --- | --- |
| 4 넣을 때 | **3** (가장 오래됨) | **3** (횟수 1회) |
| 5 넣을 때 | **4** | **4** (횟수 1회) |
| 7 넣을 때 | **1** (가장 오래됨) | **5** (횟수 1회) ← 갈림! |

```
LRU 최종 프레임: [7, 2, 5]
LFU 최종 프레임: [1, 2, 7]
```

**최종 상태가 다른데 부재 횟수만 6으로 같다.** 참조열이 한 칸만 더 길었으면 답이 갈렸다.

## LRU 추적 (마지막 사용 시점 기준)

```
1*: [1]
2*: [1,2]
3*: [1,2,3]
1 : hit  → 순서: 2, 3, 1
2 : hit  → 순서: 3, 1, 2
4*: 가장 오래된 3 제거 → [1,2,4]
1 : hit  → 순서: 2, 4, 1
2 : hit  → 순서: 4, 1, 2
5*: 가장 오래된 4 제거 → [1,2,5]
7*: 가장 오래된 1 제거 → [2,5,7]
```
**부재: 1, 2, 3, 4, 5, 7 → 6회**

## LFU 추적 (누적 횟수 기준)

```
1*: [1]           c1=1
2*: [1,2]         c2=1
3*: [1,2,3]       c3=1
1 : hit           c1=2
2 : hit           c2=2
4*: 횟수 → 1:2, 2:2, 3:1  → 최소는 3 → 버림
    [1,2,4]       c4=1
1 : hit           c1=3
2 : hit           c2=3
5*: 횟수 → 1:3, 2:3, 4:1  → 최소는 4 → 버림
    [1,2,5]       c5=1
7*: 횟수 → 1:3, 2:3, 5:1  → 최소는 5 → 버림
    [1,2,7]
```
**부재: 1, 2, 3, 4, 5, 7 → 6회**

> 이 문제는 **횟수 동점 상황이 한 번도 안 나온다.** 매번 최소가 딱 하나라 논란 없이 6회.

## ⚠️ LFU 동점 처리 - 시험 함정

**횟수가 같은 페이지가 여러 개면?** 문제가 안 정해주면 답이 갈린다.

```
관례: 동점이면 → 먼저 들어온 것(FIFO) 또는 오래된 것(LRU)을 버림
```

시험에선 동점이 안 생기게 출제하는 게 보통이지만, 생기면 문제 조건을 확인할 것.

## 다음에 푸는 기준

```
LRU  "마지막으로 언제 썼어?"  → 시계
LFU  "지금까지 몇 번 썼어?"   → 계산기

같은 답이 나올 수 있음. 하지만 우연임.
LFU 약점: 옛날에 많이 쓴 페이지가 영원히 안 나감

푸는 법은 하나 — 표 그려서 한 줄씩 추적할 것.
  LRU는 옆에 "마지막 사용 시점"
  LFU는 옆에 "누적 횟수"
머리로 하면 반드시 실수한다.
```

# 10번 - 실행 순서 (오버로딩 vs 오버라이딩)

**정답: 5 → 6 → 3 → 1 → 7 → 2**

## 핵심 - 원본 문제

```java
class Main {
    public static void main(String[] args) {   // ⑤
        Parent parent = new Child(3);          // ⑥
        System.out.println(parent.getT());     // ⑦   ← 매개변수 없음!
    }
}
```

```
⑤  main 시작
⑥  new Child(3) 객체 생성
③  Child(int x) 생성자 진입 → 첫 줄 super(x+1, x)
①  Parent(int x, int y) 실행     ← 부모 생성자가 먼저 끝나야 함
    (돌아와서 Child의 this.x = x 실행. 번호 없음)
⑦  parent.getT() 호출
②  Parent의 getT() 실행
```

**④는 실행되지 않는다.** `getT(int n)`은 매개변수가 있는 **별개 메서드**이므로.
(빈칸이 5개인 것이 ④가 안 쓰인다는 힌트다)

## 질문: ⑦이 `getT(3)`이었다면 `5 6 3 1 7 4 2`가 맞나?

> **순서 논리는 맞다. 하지만 그 코드는 컴파일이 안 된다.**

```java
Parent parent = new Child(3);   // 선언 타입은 Parent
parent.getT(3);                 // ❌ 컴파일 에러
```

**Parent 클래스에 `getT(int)`가 없다.**

```java
class Parent {
    int getT() { ... }          // ← Parent가 가진 건 이것뿐
}
class Child extends Parent {
    int getT(int n) { ... }     // ← 이건 Child만 가짐
}
```

### 핵심 규칙

> **자바는 "어떤 메서드가 있는지"를 컴파일할 때 선언 타입으로 판단한다.**

```
컴파일러: "Parent 타입 변수네. Parent에 getT(int) 있나?"
         → 없음
         → error: method getT in class Parent cannot be applied to given types
```

실제 객체가 Child든 말든 안 본다.

## 오버로딩 vs 오버라이딩 - 여기가 갈림길

`getT()`와 `getT(int n)`은 **이름만 같고 매개변수가 다른 별개 메서드**다.
→ **오버라이딩이 아니라 오버로딩이다.**

| | 오버라이딩 | 오버로딩 |
| --- | --- | --- |
| 시그니처 | **완전히 같음** | **매개변수가 다름** |
| 결정 시점 | **실행할 때** (동적 바인딩) | **컴파일할 때** (정적 바인딩) |
| 기준 | **실제 객체** 타입 | **선언된 변수** 타입 |

```
오버라이딩 → 실제 객체를 보고 실행 시점에 고름  → 부모 타입 변수로도 자식 메서드 호출됨
오버로딩   → 선언 타입을 보고 컴파일 시점에 고름 → 부모 타입 변수면 부모 것만 보임
```

**Child는 getT()를 오버라이딩한 게 아니라, getT(int)를 새로 추가한 것뿐이다.**
그래서 `Parent parent` 변수로는 접근할 길이 없다.

## 답이 `5 6 3 1 7 4 2`가 되려면

변수 선언을 바꿔야 한다:

```java
Child child = new Child(3);        // ← 선언 타입을 Child로!
System.out.println(child.getT(3)); // ⑦
```

```
⑤  main 시작
⑥  new Child(3)
③  Child(int x) 생성자 → super(x+1, x)
①  Parent(int x, int y) 실행
⑦  child.getT(3) 호출
④  getT(int n) 진입 → super.getT() 호출
②  Parent의 getT() 실행
```
**→ 5 6 3 1 7 4 2** ✅ (7개 번호 전부 등장)

## 보너스 - 출력값 함정 (필드 shadowing)

```java
class Parent { int x, y; }
class Child extends Parent { int x; }   // ← Child에 x가 또 있음!
```

**x가 두 개다.** Child의 x가 Parent의 x를 **가린다(shadowing)**.

```
new Child(3)
  super(3+1, 3) → Parent의 x=4, y=3
  this.x = 3     → Child의 x=3   (Parent의 x=4는 그대로 살아있음!)

parent.getT()  → Parent의 getT() 실행
  return x*y   → 여기서 x는 Parent의 x = 4
  → 4 * 3 = 12
```

**출력: 12** (3×3=9 아님)

> **필드는 오버라이딩이 안 된다. 가려질 뿐이다.**
> Parent의 메서드 안에서 쓰는 `x`는 항상 Parent의 x다.
> **메서드는 실제 객체를 따라가지만, 필드는 선언된 위치를 따라간다.**

`child.getT(3)`이었으면 → `super.getT() + 3` = 12 + 3 = **15**

## 다음에 푸는 기준

```
메서드(오버라이딩) → 실제 객체 따라감   (동적 바인딩)
메서드(오버로딩)   → 선언 타입 따라감   (정적 바인딩)
필드              → 선언 타입 따라감   (오버라이딩 없음, shadowing만)

객체 생성 순서:
  자식 생성자 진입 → super()로 부모 생성자 먼저 실행 → 부모 끝나고 자식 나머지

이름이 같아도 매개변수가 다르면 별개 메서드 (오버로딩)
부모 타입 변수로는 자식이 추가한 오버로딩 메서드를 호출할 수 없다
```

# 15번 - Rootkit (악성코드)

**보기:** ㄱ. Worm ㄴ. Trojan horse ㄷ. Backdoor ㄹ. Virus ㅁ. Ransomware ㅂ. Spyware ㅅ. Rootkit

**정답: ㅅ (Rootkit)**

## 지문 키워드

```
"존재를 숨기면서"           → 은폐
"무제한 접근 권한"          → root 권한
"시스템콜을 해킹"           → 커널 조작
"안티바이러스 탐지를 우회"   → 탐지 회피
```

→ **Rootkit**

**"Root(최고 관리자 권한) + Kit(도구 모음)"**의 합성어. 이름이 곧 정의다.

## 보기 용어 전체 정리

| 용어 | 한글 | 핵심 키워드 | 설명 |
| --- | --- | --- | --- |
| **Worm** | 웜 | **자가 전파**, 숙주 불필요 | **스스로** 네트워크를 타고 복제·확산. 독립 실행 파일. 사용자 조작 없이 퍼짐 |
| **Trojan horse** | 트로이 목마 | **위장**, 자기복제 X | 정상 프로그램인 척 위장. 사용자가 **직접 실행**하게 유도. **스스로 복제하지 않음** |
| **Backdoor** | 백도어 | **인증 우회**, 뒷문 | 정상 인증 절차를 건너뛰고 들어오는 통로. 개발자가 남긴 것 or 공격자가 심은 것 |
| **Virus** | 바이러스 | **숙주 필요**, 기생 | 다른 파일에 **기생**해서 복제. 숙주 파일이 실행돼야 동작 |
| **Ransomware** | 랜섬웨어 | **암호화 + 몸값** | 파일을 암호화하고 복호화 대가로 금전 요구 |
| **Spyware** | 스파이웨어 | **정보 수집** | 사용자 몰래 정보를 수집·유출 |
| **Rootkit** | 루트킷 | **은폐 + 최고권한** | 자신을 숨기며 root 권한 유지. 시스템콜 후킹으로 **탐지 회피** ← **정답** |

## 헷갈리는 것 구분

### Virus vs Worm (최다 출제)

| | Virus | Worm |
| --- | --- | --- |
| **숙주** | **필요** (파일에 기생) | **불필요** (독립 실행) |
| **전파** | 숙주 파일을 실행해야 | **스스로** 네트워크로 |
| **비유** | 감기 (사람에 붙어야 함) | 스스로 굴러다니는 벌레 |

> **"자기 스스로 복제하며 전파" → Worm**
> **"다른 파일에 기생" → Virus**

### Trojan vs Backdoor vs Rootkit

```
Trojan   = 정상인 척 "위장"해서 들어옴          (입장 방법)
Backdoor = 인증 없이 들어오는 "뒷문"            (통로)
Rootkit  = 들어와서 "숨어있으며 최고권한 유지"   (은폐 + 권한)
```

셋 다 함께 쓰이는 경우가 많지만 **역할이 다르다.**

## 다음에 푸는 기준

```
"은폐 / 탐지 우회 / 무제한 권한 / 시스템콜"   → Rootkit
"자가 복제 / 스스로 전파 / 숙주 불필요"       → Worm
"기생 / 숙주 파일 필요"                       → Virus
"위장 / 정상 프로그램인 척"                   → Trojan horse
"인증 우회 / 뒷문"                            → Backdoor
"암호화 / 몸값 / 금전 요구"                   → Ransomware
"몰래 정보 수집·유출"                         → Spyware
```

# 17번 - APT (보안 공격)

**보기:** ㄱ. 사회공학 기법 ㄴ. Adware ㄷ. MITM ㄹ. XDR ㅁ. Replace attack ㅂ. key logger attack ㅅ. APT

**정답: ㅅ (APT)**

## 지문 키워드

```
"명확한 표적을 정하여"        → 특정 타깃 (불특정 다수 X)
"지속적인 정보수집 후 공격"    → Persistent (지속)
"Zero-day 취약점"            → Advanced (지능형)
"침투, 검색, 수집, 유출 4단계" → APT의 표준 단계
```

→ **APT**

## 이름이 곧 정의다

> **A**dvanced **P**ersistent **T**hreat = **지능형 지속 위협**

| 글자 | 뜻 | 지문 대응 |
| --- | --- | --- |
| **A**dvanced | **지능형** — Zero-day 등 고도 기술 | 3번 지문 |
| **P**ersistent | **지속적** — 장기간 잠복하며 정보 수집 | 1번 지문 |
| **T**hreat | **위협** — 명확한 특정 표적 | 1번 지문 |

**APT의 4단계:** **침투 → 검색 → 수집 → 유출**

## 보기 용어 전체 정리

| 용어 | 한글 | 구분 | 핵심 키워드 | 설명 |
| --- | --- | --- | --- | --- |
| **사회공학 기법** | Social Engineering | 공격 | **사람의 심리** | 기술이 아닌 **사람**을 속임. 피싱, 보이스피싱, 어깨너머 훔쳐보기 |
| **Adware** | 애드웨어 | 악성코드 | **광고 강제 노출** | 무료 SW에 끼워져 광고를 띄움. 상대적으로 경미 |
| **MITM** | 중간자 공격 | 공격 | **통신 중간에 끼어듦** | Man-In-The-Middle. 두 통신 당사자 사이에서 **도청·변조** |
| **XDR** | 확장 탐지 대응 | ⚠️ **방어 솔루션** | **보안 제품** | Extended Detection and Response. **공격이 아님!** ← 함정 |
| **Replace attack** | (재전송 공격) | 공격 | **가로챈 정보 재전송** | 표준 용어는 **Replay attack(재전송 공격)**. 인증 정보를 가로챘다가 그대로 다시 보냄 |
| **Key logger attack** | 키로거 공격 | 공격 | **키보드 입력 기록** | 입력하는 키를 기록해 계정·카드번호 탈취 |
| **APT** | 지능형 지속 위협 | 공격 | **특정 표적 + 지속 + Zero-day** | ← **정답** |

## ⚠️ 함정 - XDR은 공격이 아니다

**보기에 방어 솔루션을 섞어놨다.** 이런 식의 함정이 자주 나온다.

```
EDR  Endpoint Detection and Response   — 단말(엔드포인트) 탐지·대응
NDR  Network Detection and Response    — 네트워크 탐지·대응
XDR  eXtended Detection and Response   — 통합 확장 탐지·대응
```

**전부 "지키는 쪽" 제품이다.** 뒤에 **"DR (Detection and Response)"이 붙으면 방어 솔루션**이라고 외워두면 된다.

## ⚠️ Replace attack 표기 주의

**표준 보안 용어는 `Replay attack`(재전송 공격)이다.** `Replace attack`은 복원 과정의 오타로 보인다.
시험에서 **`Replay attack`으로 나오면 "재전송 공격"**이다.

```
Replay attack = 인증 정보를 가로챘다가 → 나중에 그대로 다시 전송해서 인증 통과
방어법: Timestamp, Nonce(일회용 난수), Sequence Number
```

## 헷갈리는 것 구분

### APT vs 사회공학

```
사회공학 = 사람을 속이는 "기법"          (수단)
APT      = 특정 표적을 장기간 노리는 "공격 캠페인"  (전체 작전)

APT 공격이 침투 단계에서 사회공학(피싱 메일)을 "사용"한다.
→ 포함 관계. 지문에 "4단계", "지속적", "특정 표적"이 나오면 APT
```

### MITM vs Replay

```
MITM   = 통신 "중간에 끼어들어" 실시간 도청·변조
Replay = 가로챈 걸 "나중에 다시 전송"
```

## 다음에 푸는 기준

```
"특정 표적 / 지속적 / Zero-day / 침투-검색-수집-유출"  → APT
"사람의 심리 / 속임 / 피싱"                            → 사회공학
"중간에서 도청·변조"                                    → MITM
"가로챈 인증정보 재전송"                                → Replay attack
"키보드 입력 기록"                                      → Key logger
"광고 강제 노출"                                        → Adware
"~DR (EDR/NDR/XDR)"                                    → 방어 솔루션! 공격 아님
```

# 오늘의 핵심 요약

## 한 줄 정리

| 문제 | 핵심 |
| --- | --- |
| **6번** | **BCNF = 모든 결정자가 후보키.** 결정자인데 후보키 아닌 놈 있으면 → **3NF에서 멈춤** |
| **7번** | **Shortest Path First** = 최단경로 = **다익스트라** → OSPF. 이름이 곧 답 |
| **8번** | **세타 ⊃ 동등 ⊃ 자연.** 조인 = **카티션 프로덕트 + 셀렉트** |
| **9번** | **LRU는 시계, LFU는 계산기.** 답이 같아도 우연. 표 그려서 추적할 것 |
| **10번** | **오버로딩은 정적 바인딩.** 부모 타입 변수로는 자식이 추가한 메서드 호출 불가 |
| **15번** | **은폐 + 탐지우회 + 최고권한** → Rootkit (Root + Kit) |
| **17번** | **Advanced Persistent Threat.** 이름이 곧 정의. **XDR은 방어 솔루션(함정)** |

## 오늘 발견한 내 약점

```
1. 기본키 vs 후보키를 구분 못 함
   → 후보키는 "자격 있는 후보들"(여러 개 가능)
   → 기본키는 "그중 뽑힌 하나" (계산이 아니라 선택!)
   → 이게 안 되면 정규화 판별이 아예 안 됨

2. 표에서 함수 종속을 "세로로 훑어" 찾는 훈련이 안 됨
   → 6번의 강사번호 → 강좌명을 못 찾은 이유
   → 이게 BCNF 문제의 승부처

3. 용어 암기 부족 (7, 15, 17번)
   → 전부 "이름이 곧 정의"인 약어들
   → OSPF, APT, Rootkit 모두 풀네임을 풀면 답이 나옴
```

## 반드시 기억할 두문자

```
정규화:  원 - 부 - 이 - 결 - 다 - 조
        원자값 부분종속 이행종속 결정자 다치종속 조인종속
        1NF    2NF     3NF     BCNF   4NF     5NF

키:      슈퍼키 ⊃ 후보키 ⊃ 기본키
                        └→ 나머지 = 대체키

조인:    세타 ⊃ 동등 ⊃ 자연

응집도:  기 - 순 - 교 - 절 - 시 - 논 - 우  (높→낮)
        ※ 응집도는 높을수록 좋고, 결합도는 낮을수록 좋다
```
