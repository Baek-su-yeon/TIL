# Ⅰ. 이론

## 1. GoF 디자인 패턴

### 1.1 GoF 패턴이란?

GoF(Gang of Four) 디자인 패턴은 객체지향 설계에서 반복해서 나타나는 문제를 해결하기 위한 대표적인 설계 방법 23가지를 말한다.

| 분류 | 핵심 질문 | 의미 | 암기 키워드 |
|---|---|---|---|
| 생성 패턴 | 객체를 **어떻게 만들 것인가?** | 객체 생성 과정을 캡슐화하여 생성 방식의 유연성을 높임 | 생성 |
| 구조 패턴 | 객체·클래스를 **어떻게 조합할 것인가?** | 여러 객체를 연결하여 더 큰 구조를 만듦 | 조립·연결 |
| 행위 패턴 | 객체들이 **어떻게 역할을 나누고 소통할 것인가?** | 객체 간 책임, 알고리즘, 메시지 전달 방식을 정의 | 동작·소통 |

### 1.2 생성 패턴 5개

| 패턴 | 핵심 의미 | 대표적인 상황 | 암기 문장 |
|---|---|---|---|
| **Singleton** | 클래스의 인스턴스를 **단 하나만 생성**하고 공유 | 설정 관리자, 로그 관리자 | 하나만 만든다 |
| **Factory Method** | 객체 생성을 서브클래스에 맡김 | 종류에 따라 다른 객체 생성 | 만드는 방법을 자식이 결정 |
| **Abstract Factory** | 관련 있는 객체들을 **제품군 단위**로 생성 | Windows용 UI와 Mac용 UI 세트 | 관련 객체 세트를 만든다 |
| **Builder** | 복잡한 객체를 단계적으로 생성 | 옵션이 많은 객체 조립 | 순서대로 조립한다 |
| **Prototype** | 기존 객체를 복제하여 새 객체 생성 | 생성 비용이 큰 객체 | 원본을 복사한다 |

#### 생성 패턴 암기

```text
싱 팩 추 빌 프
Singleton → Factory Method → Abstract Factory → Builder → Prototype
```

> `Abstract Factory`는 여러 종류의 관련 객체를 한꺼번에 만드는 **공장 묶음**, `Factory Method`는 객체 하나의 구체적인 생성 방법을 하위 클래스가 결정하는 방식이다.

### 1.3 구조 패턴 7개

| 패턴 | 핵심 의미 | 대표적인 상황 | 암기 문장 |
|---|---|---|---|
| **Adapter** | 서로 맞지 않는 인터페이스를 변환 | 다른 규격의 클래스 연결 | 변환 어댑터 |
| **Bridge** | 기능 계층과 구현 계층을 분리 | 기능과 구현을 독립적으로 확장 | 기능과 구현의 다리 |
| **Composite** | 개별 객체와 복합 객체를 동일하게 처리 | 파일과 폴더의 트리 구조 | 부분과 전체를 동일 취급 |
| **Decorator** | 기존 객체에 기능을 동적으로 추가 | 기본 기능에 옵션을 겹쳐 추가 | 포장하여 기능 추가 |
| **Facade** | 복잡한 내부 시스템에 단순한 통합 창구 제공 | 여러 기능을 하나의 API로 제공 | 단순한 정문 |
| **Flyweight** | 공통 객체를 공유하여 메모리 절약 | 같은 문자를 수천 번 표시 | 가벼운 공유 객체 |
| **Proxy** | 실제 객체 대신 접근을 제어하는 대리 객체 사용 | 지연 생성, 원격 접근, 권한 검사 | 대리인 |

#### 구조 패턴 암기

```text
어 브 컴 데 퍼 플 프
Adapter → Bridge → Composite → Decorator → Facade → Flyweight → Proxy
```

### 1.4 행위 패턴 11개

| 패턴 | 핵심 의미 | 대표적인 상황 | 암기 문장 |
|---|---|---|---|
| **Chain of Responsibility** | 요청을 처리할 객체를 연결된 후보들이 차례로 판단 | 결재선, 이벤트 전달 | 책임을 사슬처럼 넘김 |
| **Command** | 요청을 객체로 캡슐화 | 실행 취소, 작업 큐 | 명령 자체를 객체로 |
| **Interpreter** | 문법 규칙을 정의하고 문장을 해석 | 간단한 언어·수식 해석 | 문법 해석기 |
| **Iterator** | 내부 구조를 노출하지 않고 요소를 순회 | 컬렉션 반복 | 순회자 |
| **Mediator** | 객체들이 직접 얽히지 않고 중재자를 통해 통신 | 채팅방, 관제 시스템 | 중재자가 소통 관리 |
| **Memento** | 객체의 이전 상태를 저장하고 복원 | 실행 취소, 저장 지점 | 상태 스냅샷 |
| **Observer** | 한 객체의 상태 변화 시 의존 객체에 자동 통보 | 구독 알림 | 상태 변화 알림 |
| **State** | 객체 상태에 따라 행동이 달라짐 | 자판기 상태, 문 상태 | 상태가 행동을 결정 |
| **Strategy** | 교체 가능한 알고리즘을 각각 캡슐화 | 정렬·결제 방식 선택 | 알고리즘 교체 |
| **Template Method** | 전체 처리 순서는 부모가 정하고 일부 단계는 자식이 구현 | 공통 처리 흐름 | 틀은 부모, 세부는 자식 |
| **Visitor** | 객체 구조를 바꾸지 않고 새로운 연산 추가 | 여러 객체에 새로운 처리 추가 | 방문하여 작업 추가 |

#### 행위 패턴 암기

```text
책 명 해 반 중 메 옵 상 전 템 방
책임연쇄 → 명령 → 해석자 → 반복자 → 중재자 → 메멘토
→ 옵저버 → 상태 → 전략 → 템플릿 메서드 → 방문자
```

### 1.5 기출 키워드로 바로 고르기

| 문제 속 표현 | 정답 패턴 |
|---|---|
| 인스턴스를 하나만 생성 | Singleton |
| 기능과 구현을 분리하여 독립적으로 확장 | Bridge |
| 상태 변화 시 관련 객체에 자동 통보 | Observer |
| 실제 객체 대신 접근을 제어하는 대리 객체 | Proxy |
| 호환되지 않는 인터페이스를 변환 | Adapter |
| 관련 객체들의 제품군을 생성 | Abstract Factory |
| 복잡한 시스템에 단순한 창구 제공 | Facade |
| 알고리즘을 선택·교체 | Strategy |
| 상태에 따라 동작 변경 | State |

## 2. 테스트 커버리지

### 2.1 커버리지의 의미

테스트 커버리지는 프로그램의 코드나 논리 구조가 테스트에 의해 얼마나 실행되었는지를 나타내는 기준이다.

### 2.2 주요 커버리지 비교

예시 조건문:

```java
if (A && B) {
    실행문1;
}
실행문2;
```

| 종류 | 무엇을 모두 실행해야 하는가? | 핵심 기준 |
|---|---|---|
| **문장 커버리지** | 모든 실행 문장 | 각 문장을 최소 한 번 실행 |
| **분기·결정 커버리지** | 전체 조건식의 True와 False 결과 | `A && B` 전체 결과가 참·거짓 모두 나옴 |
| **조건 커버리지** | 각 개별 조건의 True와 False | A와 B 각각 참·거짓 모두 나옴 |
| **조건/결정 커버리지** | 결정 결과 + 각 개별 조건 결과 | 분기와 조건 커버리지를 모두 만족 |
| **변경 조건/결정 커버리지(MC/DC)** | 각 조건이 독립적으로 전체 결정 결과에 영향을 줌 | 조건 하나만 바꿨을 때 결과가 바뀌는 쌍 확인 |
| **다중 조건 커버리지** | 개별 조건의 가능한 모든 조합 | A·B라면 TT, TF, FT, FF 전부 |
| **경로 커버리지** | 가능한 모든 실행 경로 | 반복문이 있으면 경로 수가 매우 커질 수 있음 |

### 2.3 자주 헷갈리는 차이

#### 문장 커버리지

```text
목표: 코드 줄을 한 번씩 실행
```

참인 경로만 실행해도 모든 문장을 지날 수 있으므로 거짓 분기가 검증되지 않을 수 있다.

#### 분기 커버리지

```text
목표: if 전체 결과의 True / False를 모두 실행
```

A와 B 각각의 참·거짓이 모두 나타난다는 보장은 없다.

#### 조건 커버리지

```text
목표: A와 B 각각의 True / False를 모두 실행
```

개별 조건은 모두 바뀌지만 전체 결정 결과가 참·거짓 모두 나온다는 보장은 없다.

> **주의:** 조건 커버리지와 분기 커버리지는 서로 보는 대상이 달라서 단순히 “조건이 무조건 분기보다 강하다”라고 외우면 위험하다.

#### MC/DC

예를 들어 `A && B`에서 A의 독립 영향을 확인하려면 B는 고정하고 A만 바꾼다.

| 테스트 | A | B | 전체 결과 |
|---|---:|---:|---:|
| 1 | T | T | T |
| 2 | F | T | F |

B를 T로 고정한 상태에서 A만 바꾸자 전체 결과가 바뀌므로 A가 독립적으로 영향을 준다는 것을 확인할 수 있다.

### 2.4 암기 기준

```text
문장: 모든 코드 줄
분기: 전체 조건식의 참/거짓
조건: 개별 조건식의 참/거짓
조건/결정: 분기 + 조건
MC/DC: 조건 하나가 독립적으로 결과를 변경
다중 조건: 모든 조건 조합
경로: 모든 실행 길
```

## 3. 응집도와 결합도

### 3.1 핵심 차이

| 구분 | 응집도 | 결합도 |
|---|---|---|
| 의미 | 한 모듈 **내부** 요소끼리 얼마나 관련 있는가 | 서로 다른 **모듈 간** 의존성이 얼마나 큰가 |
| 좋은 설계 | **높을수록 좋음** | **낮을수록 좋음** |
| 목표 | 한 모듈이 한 가지 책임에 집중 | 모듈들이 서로 독립적으로 동작 |

```text
응집도는 안쪽 결속력 → 높게
결합도는 바깥 의존성 → 낮게
```

### 3.2 응집도 7단계

#### 높은 순서: 좋은 설계 → 나쁜 설계

```text
기능적 > 순차적 > 교환적(통신적) > 절차적 > 시간적 > 논리적 > 우연적
```

암기: **기순교절시논우**

| 응집도 | 의미 | 핵심 키워드 |
|---|---|---|
| **기능적** | 모든 요소가 하나의 명확한 기능 수행 | 단일 목적 |
| **순차적** | 앞 작업의 출력이 다음 작업의 입력이 됨 | 출력 → 입력 |
| **교환적·통신적** | 여러 기능이 같은 입력 또는 출력을 사용 | 동일 데이터 |
| **절차적** | 정해진 실행 순서에 따라 기능을 수행 | 실행 순서 |
| **시간적** | 같은 시점에 실행되는 기능을 묶음 | 초기화·종료 시점 |
| **논리적** | 성격이 비슷한 기능을 하나로 묶고 매개변수로 선택 | 같은 종류, 선택 |
| **우연적** | 관련 없는 기능들이 우연히 한 모듈에 모임 | 무관한 기능 |

#### 순차적과 절차적의 차이

```text
순차적: 순서뿐 아니라 데이터가 다음 기능으로 전달됨
절차적: 실행 순서만 관련되고 데이터 흐름은 필수가 아님
```

### 3.3 결합도 6단계

#### 높은 순서: 나쁜 설계 → 좋은 설계

```text
내용 > 공통 > 외부 > 제어 > 스탬프 > 자료
```

암기: **내공외제스자**

| 결합도 | 의미 | 핵심 키워드 |
|---|---|---|
| **내용 결합도** | 다른 모듈의 내부 데이터·코드에 직접 접근 | 내부 직접 참조 |
| **공통 결합도** | 여러 모듈이 전역 데이터를 공동 사용 | 전역 변수 |
| **외부 결합도** | 외부 형식·통신 규약·장치 인터페이스 등을 공유 | 외부 환경 |
| **제어 결합도** | 제어 플래그를 전달해 상대 모듈의 흐름을 지시 | 제어 신호 |
| **스탬프 결합도** | 배열·구조체·객체 등 자료구조 전체를 전달 | 자료구조 전체 |
| **자료 결합도** | 필요한 단순 데이터 값만 매개변수로 전달 | 필요한 값만 |

#### 스탬프와 자료 결합도의 차이

```text
스탬프: 학생 객체 전체를 넘기고 그중 이름만 사용
자료: 이름 문자열만 넘김
```

### 3.4 한 줄 정리

```text
좋은 모듈 = 내부는 끈끈하게(고응집), 외부와는 느슨하게(저결합)
```

## 4. 서브넷 마스크 총정리

### 4.1 서브넷 마스크의 의미

IP 주소에서 **네트워크 부분**과 **호스트 부분**을 구분하는 32비트 값이다.

```text
서브넷 마스크의 1 → 네트워크 비트
서브넷 마스크의 0 → 호스트 비트
```

예시:

```text
IP 주소      192.168.10.130
서브넷 마스크 255.255.255.192
CIDR         /26
```

`255.255.255.192`를 2진수로 바꾸면 다음과 같다.

```text
11111111.11111111.11111111.11000000
```

1이 26개이므로 `/26`, 0이 6개이므로 호스트 비트는 6개이다.

### 4.2 자주 나오는 마스크 표

| 마지막 옥텟 | 2진수 | CIDR | 블록 크기 | 호스트 비트 | 사용 가능한 호스트 수 |
|---:|---|---:|---:|---:|---:|
| 0 | 00000000 | /24 | 256 | 8 | 254 |
| 128 | 10000000 | /25 | 128 | 7 | 126 |
| 192 | 11000000 | /26 | 64 | 6 | 62 |
| 224 | 11100000 | /27 | 32 | 5 | 30 |
| 240 | 11110000 | /28 | 16 | 4 | 14 |
| 248 | 11111000 | /29 | 8 | 3 | 6 |
| 252 | 11111100 | /30 | 4 | 2 | 2 |

```text
사용 가능한 호스트 수 = 2^(호스트 비트 수) - 2
```

2를 빼는 이유:

1. 모든 호스트 비트가 0인 주소는 **네트워크 주소**
2. 모든 호스트 비트가 1인 주소는 **브로드캐스트 주소**

> 일반적인 IPv4 서브넷 문제 기준 공식이다. `/31`, `/32` 같은 특수 용도는 보통 기사 계산 문제에서 제외한다.

### 4.3 블록 크기로 푸는 방법

```text
블록 크기 = 256 - 서브넷 마스크의 해당 옥텟 값
```

예: `255.255.255.192`

```text
블록 크기 = 256 - 192 = 64
범위 시작값 = 0, 64, 128, 192
```

따라서 마지막 옥텟 구간은 다음과 같다.

| 네트워크 주소 | 사용 가능한 호스트 | 브로드캐스트 주소 |
|---:|---:|---:|
| 0 | 1~62 | 63 |
| 64 | 65~126 | 127 |
| 128 | 129~190 | 191 |
| 192 | 193~254 | 255 |

### 4.4 예제 1: 네트워크 주소와 호스트 수

```text
IP: 223.13.234.132
Mask: 255.255.255.192
```

1. 블록 크기: `256 - 192 = 64`
2. 구간: 0, 64, 128, 192
3. 132는 `128~191` 구간에 포함
4. 네트워크 주소: `223.13.234.128`
5. 브로드캐스트 주소: `223.13.234.191`
6. 사용 가능 범위: `223.13.234.129 ~ 223.13.234.190`
7. 호스트 수: `2^6 - 2 = 62`

### 4.5 예제 2: 마스크가 세 번째 옥텟에 걸린 경우

```text
IP: 192.168.35.10
Mask: 255.255.252.0
```

세 번째 옥텟에서 계산한다.

```text
블록 크기 = 256 - 252 = 4
구간 시작 = 0, 4, 8, ..., 32, 36, ...
35는 32~35 구간에 포함
```

따라서:

```text
네트워크 주소: 192.168.32.0
브로드캐스트: 192.168.35.255
사용 가능 호스트: 192.168.32.1 ~ 192.168.35.254
```

### 4.6 AND 연산으로 네트워크 주소 구하기

```text
네트워크 주소 = IP 주소 AND 서브넷 마스크
```

예시 마지막 옥텟:

```text
IP 130   = 10000010
Mask 192 = 11000000
AND      = 10000000 = 128
```

따라서 네트워크 주소의 마지막 옥텟은 128이다.

### 4.7 브로드캐스트 주소 구하기

```text
브로드캐스트 주소 = 네트워크 주소의 호스트 비트를 모두 1로 설정
```

또는 블록 크기 방식으로:

```text
브로드캐스트 = 다음 네트워크 주소 - 1
```

### 4.8 같은 네트워크인지 확인

두 IP에 각각 같은 서브넷 마스크를 AND 연산한다.

```text
IP A AND Mask = 네트워크 A
IP B AND Mask = 네트워크 B
```

두 네트워크 주소가 같으면 같은 서브넷이다.

### 4.9 풀이 순서

```text
① 마스크에서 계산할 옥텟 찾기
② 블록 크기 = 256 - 마스크 값
③ IP가 속한 구간 찾기
④ 구간 시작 = 네트워크 주소
⑤ 다음 구간 시작 - 1 = 브로드캐스트 주소
⑥ 호스트 수 = 2^호스트 비트 - 2
```

## 5. 무결성 제약조건

### 5.1 무결성이란?

데이터베이스에 저장된 데이터의 **정확성, 일관성, 유효성**을 유지하기 위한 규칙이다.

| 무결성 | 의미 | 대표 제약 | 예시 |
|---|---|---|---|
| **도메인 무결성** | 속성 값은 정해진 자료형·범위·형식을 따라야 함 | 데이터 타입, CHECK, DEFAULT, NOT NULL | 나이는 0 이상 정수 |
| **개체 무결성** | 기본키는 각 행을 유일하게 식별해야 함 | PRIMARY KEY, UNIQUE + NOT NULL | 학번은 중복·NULL 불가 |
| **참조 무결성** | 외래키는 참조 대상 기본키 값과 일치하거나 NULL이어야 함 | FOREIGN KEY | 존재하지 않는 부서번호 입력 불가 |
| **사용자 정의 무결성** | 업무 규칙에 따라 사용자가 정의한 조건을 지켜야 함 | CHECK, 트리거, 애플리케이션 규칙 | 출고일은 주문일보다 빠를 수 없음 |
| **키 무결성** | 후보키 값은 릴레이션에서 유일해야 함 | UNIQUE | 주민번호 중복 불가 |
| **NULL 무결성** | 특정 속성은 NULL을 허용하지 않음 | NOT NULL | 이름 필수 입력 |

### 5.2 시험에서 가장 자주 나오는 3개

```text
도메인: 컬럼 값의 범위·형식
개체: 기본키는 중복 X, NULL X
참조: 외래키는 참조 테이블에 존재하거나 NULL
```

### 5.3 참조 무결성 관련 동작

부모 행이 수정·삭제될 때 외래키 처리 방식:

| 방식 | 의미 |
|---|---|
| RESTRICT / NO ACTION | 참조 중이면 부모 행 삭제·수정 거부 |
| CASCADE | 부모 변경을 자식에게 연쇄 반영 |
| SET NULL | 자식 외래키를 NULL로 변경 |
| SET DEFAULT | 자식 외래키를 기본값으로 변경 |

## 6. CPU 스케줄링

### 6.1 선점형과 비선점형

| 구분 | 의미 | 대표 알고리즘 |
|---|---|---|
| **비선점형** | 프로세스가 CPU를 받으면 종료·대기 전까지 빼앗지 않음 | FCFS, SJF, HRN, 비선점 Priority |
| **선점형** | 실행 중이라도 다른 프로세스가 CPU를 빼앗을 수 있음 | RR, SRT, 선점 Priority, 다단계 큐 |

### 6.2 주요 스케줄링 알고리즘

| 알고리즘 | 선택 기준 | 선점 여부 | 장점 | 단점·키워드 |
|---|---|---|---|---|
| **FCFS** | 먼저 도착한 순서 | 비선점 | 단순하고 공평 | 긴 작업 뒤에 짧은 작업이 대기하는 호위 효과 |
| **SJF** | 실행 시간이 가장 짧은 작업 | 비선점 | 평균 대기시간 최소화에 유리 | 긴 작업 기아 가능, 실행시간 예측 필요 |
| **SRT** | 남은 실행시간이 가장 짧은 작업 | 선점 | 짧은 작업의 응답성 좋음 | 새 짧은 작업 도착 시 CPU 선점 |
| **RR** | 각 프로세스에 시간 할당량을 순환 배정 | 선점 | 시분할, 응답시간 개선 | 시간 할당량이 너무 작으면 문맥 교환 증가 |
| **Priority** | 우선순위가 높은 작업 | 둘 다 가능 | 중요 작업 우선 처리 | 낮은 우선순위 기아, 에이징으로 해결 |
| **HRN** | 응답률이 높은 작업 | 비선점 | SJF의 기아 문제 완화 | 계산 필요 |
| **다단계 큐** | 프로세스 종류별 큐 분리 | 보통 선점 | 유형별 정책 적용 | 큐 간 이동이 제한적 |
| **다단계 피드백 큐** | 실행 특성에 따라 큐 사이 이동 | 선점 | 적응적, 대화형 작업에 유리 | 구현 복잡 |

### 6.3 핵심 공식

```text
반환시간 = 완료시간 - 도착시간
대기시간 = 반환시간 - 실행시간
응답시간 = 최초 CPU 실행시간 - 도착시간
```

#### HRN 응답률

```text
응답률 = (대기시간 + 실행시간) / 실행시간
       = 1 + 대기시간 / 실행시간
```

응답률이 가장 높은 프로세스를 먼저 실행한다.

### 6.4 SJF와 SRT 비교

| 구분 | SJF | SRT |
|---|---|---|
| 기준 | 총 실행시간이 가장 짧음 | 남은 실행시간이 가장 짧음 |
| 선점 | 비선점 | 선점 |
| 실행 중 더 짧은 작업 도착 | 기존 작업 계속 | 새 작업이 CPU를 빼앗음 |

### 6.5 RR 계산 순서

```text
① 도착한 프로세스를 준비 큐에 넣음
② 큐의 맨 앞 프로세스를 시간 할당량만큼 실행
③ 끝났으면 제거
④ 안 끝났으면 남은 시간을 기록하고 큐 뒤로 이동
⑤ 새로 도착한 프로세스도 큐 뒤에 추가
```

#### 시간 할당량의 영향

```text
매우 큼 → FCFS와 비슷
매우 작음 → 응답성은 좋아질 수 있지만 문맥 교환 비용 증가
```

### 6.6 암기 문장

```text
FCFS: 먼저 온 순서
SJF: 가장 짧은 작업, 안 뺏김
SRT: 남은 시간이 가장 짧은 작업, 뺏김
RR: 정해진 시간씩 돌아가며 실행
Priority: 높은 우선순위 먼저
HRN: 오래 기다린 작업을 보정
```

## 7. 데이터베이스 용어

### 7.1 릴레이션 구조 용어

| 용어 | 의미 | 쉽게 말하면 |
|---|---|---|
| **릴레이션(Relation)** | 행과 열로 구성된 논리적 테이블 | 테이블 |
| **튜플(Tuple)** | 릴레이션의 한 행 | 레코드 한 건 |
| **어트리뷰트(Attribute)** | 릴레이션의 한 열 | 컬럼, 속성 |
| **카디널리티(Cardinality)** | 튜플의 개수 | 행 개수 |
| **디그리·차수(Degree)** | 어트리뷰트의 개수 | 열 개수 |
| **도메인(Domain)** | 한 속성이 가질 수 있는 값의 범위 | 값의 자료형·범위 |

```text
행 하나 = 튜플
행 개수 = 카디널리티
열 하나 = 어트리뷰트
열 개수 = 디그리
```

### 7.2 스키마와 인스턴스

| 용어 | 의미 | 변화 여부 | 비유 |
|---|---|---|---|
| **스키마(Schema)** | 데이터베이스의 구조와 제약조건에 대한 정의 | 비교적 잘 변하지 않음 | 설계도 |
| **인스턴스(Instance)** | 특정 시점에 실제 저장된 데이터 전체 | 삽입·수정·삭제 때 변함 | 현재 채워진 내용 |

```text
스키마 = 틀
인스턴스 = 지금 들어 있는 값
```

### 7.3 스키마 3계층

| 단계 | 의미 | 관점 |
|---|---|---|
| **외부 스키마** | 사용자·응용 프로그램별로 보는 데이터 구조 | 사용자 뷰 |
| **개념 스키마** | 데이터베이스 전체의 논리적 구조 | 조직 전체 |
| **내부 스키마** | 실제 저장 장치에 저장되는 물리적 구조 | 저장 방식 |

```text
외부 = 사용자별 화면
개념 = 전체 논리 설계
내부 = 물리 저장
```

### 7.4 키 용어

| 키 | 의미 |
|---|---|
| **슈퍼키** | 튜플을 유일하게 식별할 수 있는 속성 집합 |
| **후보키** | 최소성을 만족하는 슈퍼키 |
| **기본키** | 후보키 중 대표로 선택한 키 |
| **대체키** | 기본키로 선택되지 않은 나머지 후보키 |
| **외래키** | 다른 릴레이션의 기본키를 참조하는 속성 |
| **복합키** | 둘 이상의 속성으로 구성된 키 |

#### 키의 두 성질

```text
유일성: 각 튜플을 구별할 수 있음
최소성: 꼭 필요한 최소 속성만 포함
```

### 7.5 예시

| 학번 | 이름 | 학과 |
|---|---|---|
| 1001 | 홍길동 | 컴퓨터 |
| 1002 | 김철수 | 전자 |
| 1003 | 이영희 | 컴퓨터 |

```text
튜플: 각 학생 한 행
어트리뷰트: 학번, 이름, 학과
카디널리티: 3
디그리: 3
학과의 도메인: 학교에서 허용한 학과명 집합
스키마: 학생(학번, 이름, 학과)
인스턴스: 표에 현재 저장된 세 학생 데이터
```

## 8. 오류 검출·정정 방식

### 8.1 FEC와 BEC

| 구분 | FEC | BEC |
|---|---|---|
| 전체 이름 | Forward Error Correction | Backward Error Correction |
| 한글 | 전진 오류 수정 | 후진 오류 수정 |
| 처리 방식 | 수신 측이 부가 정보를 이용해 직접 오류 정정 | 오류를 검출하고 송신 측에 재전송 요청 |
| 재전송 | 필요 없음 | 필요함 |
| 대표 | Hamming, Reed-Solomon 등 | ARQ 방식 |
| 적합 상황 | 재전송이 어렵거나 지연이 큰 통신 | 재전송이 가능한 통신 |

```text
FEC = 앞으로 계속 진행하며 수신 측에서 고침
BEC = 뒤로 돌아가 송신 측에 다시 보내 달라고 요청
```

### 8.2 주요 오류 검출·정정 기법

| 기법 | 역할 | 원리 | 특징 |
|---|---|---|---|
| **Parity Bit** | 오류 검출 | 1의 개수를 짝수 또는 홀수로 맞추는 비트 추가 | 단순, 짝수 개 비트 오류를 놓칠 수 있음 |
| **Hamming Code** | 오류 검출 + 정정 | 여러 검사 비트로 오류 위치 계산 | 일반적으로 1비트 오류 정정 가능, FEC 대표 |
| **CRC** | 오류 검출 | 생성 다항식으로 나눈 나머지를 전송 | 연속 오류 검출에 강함 |
| **Checksum** | 오류 검출 | 데이터 단위를 합산한 검사값 사용 | TCP/IP 등에서 사용 |
| **LRC** | 오류 검출 | 여러 문자 블록의 세로 방향 패리티 | 블록 단위 검사 |
| **VRC** | 오류 검출 | 문자마다 세로 방향 패리티 비트 추가 | 단일 문자 단위 패리티 |

> Parity, CRC, Checksum은 그 자체로 오류를 **고치는 방식이 아니라 검출하는 기법**이다. BEC에서는 이런 검출 결과를 바탕으로 재전송을 요청할 수 있다.

### 8.3 패리티 비트

#### 짝수 패리티

데이터와 패리티 비트를 합친 전체 1의 개수를 짝수로 만든다.

```text
데이터 1011001 → 1의 개수 4개
짝수 패리티 비트 = 0
```

#### 홀수 패리티

전체 1의 개수를 홀수로 만든다.

```text
데이터 1011001 → 1의 개수 4개
홀수 패리티 비트 = 1
```

### 8.4 Hamming Code

검사 비트를 1, 2, 4, 8처럼 **2의 거듭제곱 위치**에 배치한다.

검사 비트 수 `r`은 보통 다음 조건을 만족해야 한다.

```text
2^r >= m + r + 1
m = 데이터 비트 수
r = 검사 비트 수
```

핵심:

```text
검사 결과를 조합하면 오류가 발생한 비트 위치를 알아낼 수 있음
→ 해당 비트를 반전하여 정정
```

### 8.5 CRC

```text
① 송신 데이터 뒤에 0을 붙임
② 생성 다항식으로 XOR 나눗셈
③ 나머지를 데이터 뒤에 붙여 전송
④ 수신 측이 같은 다항식으로 다시 나눔
⑤ 나머지가 0이 아니면 오류로 판단
```

암기:

```text
CRC = 다항식 나눗셈 + 나머지
```

### 8.6 BEC의 ARQ 유형

| ARQ 방식 | 동작 |
|---|---|
| **Stop-and-Wait ARQ** | 한 프레임 전송 후 ACK를 기다림 |
| **Go-Back-N ARQ** | 오류 프레임부터 이후 프레임을 모두 재전송 |
| **Selective Repeat ARQ** | 오류가 난 프레임만 선택적으로 재전송 |

```text
Go-Back-N = 뒤의 것까지 다시
Selective Repeat = 틀린 것만 다시
```

## 9. 접근 통제 방식과 ARP·RARP

## 9.1 접근 통제 방식

| 방식 | 전체 이름 | 권한 결정 주체 | 핵심 기준 | 대표 환경 |
|---|---|---|---|---|
| **DAC** | Discretionary Access Control | 자원 소유자 | 소유자가 임의로 권한 부여·회수 | 일반 파일 시스템 |
| **MAC** | Mandatory Access Control | 중앙 관리자·보안 정책 | 보안 등급과 허가 등급 | 군사·기밀 환경 |
| **RBAC** | Role-Based Access Control | 조직의 역할 체계 | 사용자 역할·직무 | 기업 업무 시스템 |

### DAC

```text
파일 주인이 다른 사용자에게 읽기·쓰기 권한을 줌
```

- 유연함
- 소유자의 재량이 큼
- 권한 전파로 보안이 약해질 수 있음

### MAC

```text
객체의 보안 등급과 사용자의 인가 등급을 비교
```

- 중앙 정책이 강제됨
- 사용자가 임의로 권한을 넘길 수 없음
- 보안성이 높고 유연성은 낮음

### RBAC

```text
사용자 → 역할 → 권한
```

- 사용자마다 권한을 일일이 주지 않음
- 인사 이동 시 역할만 변경하면 됨
- 대규모 조직에서 관리가 편리함

### 암기

```text
DAC = 소유자의 재량
MAC = 관리 정책이 강제
RBAC = 역할에 권한 부여
```

## 9.2 ARP와 RARP

> ARP와 RARP는 접근 통제 방식이 아니라 **주소 변환 프로토콜**이다.

| 프로토콜 | 전체 이름 | 변환 방향 | 암기 |
|---|---|---|---|
| **ARP** | Address Resolution Protocol | IP 주소 → MAC 주소 | IP로 물리 주소 찾기 |
| **RARP** | Reverse Address Resolution Protocol | MAC 주소 → IP 주소 | ARP의 반대 |

```text
ARP: 이 IP를 가진 장치의 MAC 주소가 무엇인가?
RARP: 이 MAC 주소를 가진 장치가 사용할 IP 주소가 무엇인가?
```

### ARP 동작 흐름

```text
① 송신자가 대상 IP의 MAC 주소를 모름
② 같은 네트워크에 ARP Request를 브로드캐스트
③ 해당 IP를 가진 장치가 자신의 MAC 주소로 응답
④ 송신자가 ARP 캐시에 IP-MAC 대응 정보를 저장
```

### RARP 참고

RARP는 장치가 자신의 MAC 주소만 알고 있을 때 IP 주소를 얻기 위한 초기 방식이다. 현재는 더 많은 설정 정보를 제공하는 BOOTP와 DHCP가 주로 사용된다.

### IP 주소와 MAC 주소의 차이

| 구분 | IP 주소 | MAC 주소 |
|---|---|---|
| 의미 | 논리적 주소 | 물리적 주소 |
| 사용 계층 | 네트워크 계층 | 데이터 링크 계층 |
| 목적 | 네트워크 간 경로 선택 | 같은 네트워크 내 장치 식별 |
| 예 | 192.168.0.10 | 00:1A:2B:3C:4D:5E |

# Ⅱ. 코드

## 10. Java: 오버로딩과 오버라이딩

### 10.1 핵심 비교

| 구분 | 오버로딩(Overloading) | 오버라이딩(Overriding) |
|---|---|---|
| 의미 | 같은 이름의 메서드를 여러 형태로 정의 | 부모 메서드를 자식이 재정의 |
| 발생 위치 | 보통 같은 클래스 | 상속 관계의 부모·자식 클래스 |
| 메서드 이름 | 같음 | 같음 |
| 매개변수 | 개수·타입·순서가 달라야 함 | 같아야 함 |
| 반환형 | 반환형만 다르게 해서는 불가 | 같거나 공변 반환형 |
| 결정 시점 | 컴파일 시점 | 실행 시점 |
| 기준 | 참조 변수의 선언 타입과 전달 인수 | 실제 생성된 객체 타입 |
| 바인딩 | 정적 바인딩 | 동적 바인딩 |

```text
오버로딩 = 이름은 같고 입력이 다름
오버라이딩 = 부모 기능을 자식이 다시 정의
```

### 10.2 오버로딩 예제

```java
class Calculator {
    int add(int a, int b) {
        return a + b;
    }

    double add(double a, double b) {
        return a + b;
    }

    int add(int a, int b, int c) {
        return a + b + c;
    }
}
```

메서드 이름은 모두 `add`지만 매개변수 목록이 다르므로 오버로딩이다.

#### 오버로딩이 아닌 경우

```java
int test(int a) { return a; }
double test(int a) { return a; } // 컴파일 오류
```

반환형만 다른 것은 호출 시 구분할 수 없으므로 오버로딩이 아니다.

### 10.3 오버라이딩 예제

```java
class Parent {
    void print() {
        System.out.println("Parent");
    }
}

class Child extends Parent {
    @Override
    void print() {
        System.out.println("Child");
    }
}

public class Main {
    public static void main(String[] args) {
        Parent ref = new Child();
        ref.print();
    }
}
```

출력:

```text
Child
```

이유:

```text
선언 타입: Parent
실제 객체: Child
오버라이딩 메서드 호출 기준: 실제 객체
```

### 10.4 오버로딩과 오버라이딩이 동시에 나오는 경우

```java
class A {
    String f(Object x) { return "A-Object"; }
    String f(String x) { return "A-String"; }
}

class B extends A {
    @Override
    String f(Object x) { return "B-Object"; }
}

A ref = new B();
Object value = "hello";
System.out.println(ref.f(value));
```

판단 순서:

```text
① 오버로딩 선택: 컴파일 시점
   ref의 선언 타입은 A, value의 선언 타입은 Object
   → A의 f(Object) 선택

② 오버라이딩 선택: 실행 시점
   실제 객체는 B
   → B가 재정의한 f(Object) 실행
```

출력:

```text
B-Object
```

### 10.5 static 메서드 주의

```java
class Parent {
    static String name() { return "P"; }
}

class Child extends Parent {
    static String name() { return "C"; }
}

Parent ref = new Child();
System.out.println(ref.name());
```

출력:

```text
P
```

`static` 메서드는 객체가 아니라 클래스에 속한다. 따라서 오버라이딩이 아니라 **메서드 숨김**이며 선언 타입을 기준으로 결정된다.

```text
인스턴스 오버라이딩 메서드 → 실제 객체 기준
static 메서드 → 선언 타입 기준
```

### 10.6 생성자는 오버라이딩되지 않는다

- 생성자는 상속되지 않는다.
- 같은 클래스 안에서 매개변수를 다르게 한 생성자는 **오버로딩**할 수 있다.
- 자식 생성자는 `super(...)`로 부모 생성자를 호출한다.

## 11. C: 포인터, 연결 리스트, `->` 연산자

### 11.1 포인터의 의미

포인터는 다른 변수의 **메모리 주소**를 저장하는 변수이다.

```c
int a = 10;
int *p = &a;
```

| 표현 | 의미 | 결과 |
|---|---|---|
| `a` | 변수 a의 값 | 10 |
| `&a` | 변수 a의 주소 | 주소값 |
| `p` | p에 저장된 주소 | `&a`와 같음 |
| `*p` | p가 가리키는 주소의 값 | 10 |

```text
& = 주소를 구함
* = 주소를 따라가 값을 읽거나 수정
```

### 11.2 포인터로 값 변경

```c
int a = 10;
int *p = &a;
*p = 20;
```

`*p`는 a 자체를 가리키므로 a의 값도 20으로 바뀐다.

### 11.3 배열과 포인터

```c
int arr[3] = {10, 20, 30};
int *p = arr;
```

다음 표현은 같다.

```c
arr[i] == *(arr + i)
p[i]   == *(p + i)
```

예:

```c
*(p + 1) == 20
```

### 11.4 이중 포인터

```c
int a = 10;
int *p = &a;
int **pp = &p;
```

| 표현 | 의미 |
|---|---|
| `p` | a의 주소 |
| `*p` | a의 값 |
| `pp` | p의 주소 |
| `*pp` | p에 저장된 값, 즉 a의 주소 |
| `**pp` | a의 값 |

### 11.5 구조체 멤버 접근: `.`과 `->`

```c
struct Student {
    int score;
};
```

#### 구조체 변수

```c
struct Student s;
s.score = 90;
```

#### 구조체 포인터

```c
struct Student *p = &s;
p->score = 90;
```

다음 두 표현은 같다.

```c
p->score == (*p).score
```

```text
.  = 구조체 변수의 멤버 접근
-> = 구조체 포인터가 가리키는 객체의 멤버 접근
```

### 11.6 연결 리스트의 노드

```c
typedef struct Node {
    int data;
    struct Node *next;
} Node;
```

노드 하나는 다음 두 정보를 가진다.

```text
data: 현재 노드의 값
next: 다음 노드의 주소
```

그림:

```text
head
 ↓
[10 | next] → [20 | next] → [30 | NULL]
```

### 11.7 노드 생성

```c
Node *newNode = (Node *)malloc(sizeof(Node));
newNode->data = 10;
newNode->next = NULL;
```

| 코드 | 의미 |
|---|---|
| `malloc(sizeof(Node))` | Node 하나를 저장할 메모리 동적 할당 |
| `newNode->data = 10` | 새 노드 값 설정 |
| `newNode->next = NULL` | 아직 다음 노드가 없음 |

사용이 끝난 노드는 다음과 같이 해제한다.

```c
free(newNode);
```

### 11.8 맨 앞에 노드 삽입

```c
Node *newNode = (Node *)malloc(sizeof(Node));
newNode->data = value;
newNode->next = head;
head = newNode;
```

순서가 중요하다.

```text
① 새 노드의 next가 기존 head를 가리킴
② head를 새 노드로 변경
```

기존:

```text
head → [20] → [30]
```

10 삽입 후:

```text
head → [10] → [20] → [30]
```

### 11.9 맨 뒤에 노드 삽입

```c
Node *curr = head;
while (curr->next != NULL) {
    curr = curr->next;
}
curr->next = newNode;
```

`curr`를 마지막 노드까지 이동시킨 뒤 마지막 노드의 `next`에 새 노드 주소를 넣는다.

### 11.10 리스트 순회

```c
Node *curr = head;

while (curr != NULL) {
    printf("%d ", curr->data);
    curr = curr->next;
}
```

```text
curr->data: 현재 값 읽기
curr = curr->next: 다음 노드로 이동
curr == NULL: 리스트 끝
```

### 11.11 중간 노드 제거

구조:

```text
prev → curr → next
```

`curr`를 리스트에서 제외하려면:

```c
prev->next = curr->next;
free(curr);
```

연결 변화:

```text
변경 전: prev → curr → next
변경 후: prev ─────→ next
```

> 먼저 연결을 바꾸고, 그다음 제거한 노드를 `free`해야 한다.

### 11.12 특정 노드를 맨 앞으로 이동

```c
prev->next = curr->next;  // 기존 위치에서 떼기
curr->next = head;        // 기존 head 앞에 연결
head = curr;              // head 갱신
```

```text
노드 이동 = 기존 위치에서 떼기 + 새 위치에 붙이기
```

### 11.13 다중 `->` 해석

```c
head->next->data
```

읽는 순서:

```text
head가 가리키는 노드
→ 그 노드의 next가 가리키는 다음 노드
→ 그 다음 노드의 data
```

```c
head->next->next->data
```

head에서 두 칸 이동한 노드의 값을 의미한다.

### 11.14 연결 추적 문제 풀이법

코드를 머릿속으로만 계산하지 말고 노드별 연결을 표로 적는다.

| 노드 | data | next가 가리키는 노드 |
|---|---:|---|
| a | 1 | b |
| b | 2 | c |
| c | 3 | NULL |

그리고 `head`부터 화살표를 따라간다.

```text
① 각 노드의 값 기록
② 각 노드의 next 변경을 한 줄씩 반영
③ 마지막 head 위치 확인
④ head부터 NULL까지 화살표를 따라 출력
```

### 11.15 포인터 문제 핵심 등식

```c
p[i]          == *(p + i)
p->member     == (*p).member
arr           == &arr[0]      // 대부분의 식에서 첫 요소 주소로 변환
head->next    // 다음 노드 주소
head->next->data // 다음 노드의 값
```
